# Починить падающие тесты

Цель: найти и исправить падающие тесты после реализации задачи, не нарушая логику реализации.

## Аргументы
- `$ARGUMENTS` — номер задачи. Может быть не передан.

## Определение задачи

1. Задачи могут находиться в `ai-process/tasks/` или `ai-process/tasks-done/`
2. Формат: `{папка}/{numberOfPlan}-{description}/`
3. Поиск папки задачи:
   - Сначала искать в `ai-process/tasks/`
   - Если не найдено — искать в `ai-process/tasks-done/`
4. Если номер НЕ передан — работаем с последней по номеру задачей среди обеих папок

## Предусловия

**Обязательно прочитать перед началом работы:**
1. `plan.md` в папке задачи — понять что было реализовано
2. `analyze.md` в папке задачи — для понимания архитектуры и контекста

## Порядок работы

### 1. Запуск тестов
- Запустить все тесты: `npm test`
- Зафиксировать список падающих тестов и текст ошибок

### 2. Анализ причин
Для каждого падающего теста определить причину:
- **Тест устарел** — реализация изменилась (новые аргументы, другой формат вывода, переименование)
- **Тест некорректен** — тест написан неправильно (неверные ожидания, неправильные входные данные)
- **Баг в реализации** — тест выявил реальную ошибку в коде задачи

### 3. Устранение
В зависимости от причины:

**Тест устарел или некорректен:**
- Обновить тест под текущую реализацию
- Сохранить намерение теста (что он проверяет), менять только механику

**Баг в реализации:**
- Сообщить пользователю о найденном баге
- Исправить баг в коде реализации
- Не «заглушать» тест — баг должен быть именно починен

### 4. Проверка
- Повторно запустить тесты
- Убедиться что все ранее падавшие тесты теперь проходят
- Убедиться что ранее проходившие тесты не сломались (регрессия)

## Ограничения

- **Не удалять тесты** — только чинить или обновлять
- **Не отключать тесты** через `.skip` или `xit` без явного согласия пользователя
- **Не менять логику реализации** ради прохождения тестов, если это противоречит плану — сообщить пользователю
- Если причина падения неоднозначна — **спросить пользователя** прежде чем вносить изменения

## Уточнения
- fix-tests — опциональный этап после do (analyze → plan → do → fix-tests)
- Не коммитить изменения автоматически — коммит только по запросу пользователя
- Если тест выявил реальный баг — это важная находка, явно сообщить пользователю
