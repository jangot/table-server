# Составить план работ по существующему анализу без внесения изменений

Цель: на основе analyze.md создать детальный план реализации. Код не писать, файлы проекта не менять.

## Аргументы
- `$ARGUMENTS` — номер задачи. Может быть не передан.

## Определение задачи

1. Задачи находятся в `ai-process/tasks/` или `ai-process/tasks-done/`
2. Формат: `{папка}/{numberOfPlan}-{description}/`
3. Если номер НЕ передан — работаем с последней по номеру задачей среди обеих папок

## Предусловия

**Обязательно прочитать перед составлением плана:**
1. `analyze.md` в папке задачи — если файла нет, сообщить пользователю и предложить запустить `/analyze`
2. `description.md` в папке задачи (если есть) — дополнительный контекст от пользователя

**Если пользователь передал описание задачи в сообщении** — использовать его как основное задание. Иначе — брать из `description.md`.

## Структура выходного файла

Результат: `ai-process/tasks/{numberOfPlan}-{description}/plan.md`

План должен быть **на русском** и следовать этой структуре:

```markdown
# План реализации: {название задачи}

{Краткое описание цели — 2-3 предложения}

## 1. {Название шага}

**Файл (создать/изменить):** `src/.../file.ts`

{Описание что нужно сделать}

```typescript
// Пример кода
```

## 2. {Следующий шаг}
...

## N. Тесты

**Файл (создать):** `test/{module-name}.test.ts` или рядом с исходником

Сценарии:
- Happy path: ...
- Ошибки: ...

## Сводка файлов

| Действие | Файл |
|----------|------|
| Создать  | `src/.../file.ts` |
| Изменить | `src/.../existing.ts` |

## Ссылки
- analyze.md текущей задачи
```

## Требования к плану

- Каждый шаг должен содержать **конкретные пути файлов** (создать/изменить)
- Каждый шаг должен содержать **примеры кода** на TypeScript
- Обязательно включать шаг с **тестами** — покрывать основные сценарии (happy path + ключевые ошибки)
- Завершать план **сводной таблицей файлов** — для использования как чеклист на этапе `/do`

## Большие задачи

Если задача слишком большая для одного плана, разбить на части:
- `plan.md` — первая часть
- `plan-part2.md` — вторая часть
- И т.д.

В конце каждой части указать, что продолжение — в следующем файле.

## Уточнения
- Ничего реализовывать не надо — только план
- Обновлять `analyze.md` текущей задачи не нужно
- plan — второй этап конвейера (analyze → plan → do)
