# Анализ логики и архитектуры по существующей кодовой базе для внесения изменений

Цель: проанализировать код и создать файл с описанием для дальнейшей работы (этапов plan и do).

## Аргументы
- Первый аргумент `$ARGUMENTS[0]` — номер задачи (numberOfPlan). Может быть не передан.
- Остальные аргументы начиная с `$ARGUMENTS[1]` — короткое описание задачи (description). Может быть не передано.

## Определение папки задачи

1. Все артефакты задач находятся в `ai-process/tasks/`
2. Формат имени папки: `ai-process/tasks/{numberOfPlan}-{description}/`
3. Нумерация — трёхзначная с ведущими нулями: 001, 002, ..., 042 и т.д.

**Определение numberOfPlan:**
- Если передан — использовать его
- Если НЕ передан — посмотреть существующие папки в `ai-process/tasks/` и `ai-process/tasks-done/`, взять максимальный номер

**Определение description (имя папки):**
- Если передано — использовать как часть имени папки (kebab-case, на английском)
- Если НЕ передано — определить по контексту задачи или спросить у пользователя

**Если папка уже существует:**
- Проверить наличие `description.md` — прочитать его для контекста задачи
- Проверить наличие `analyze.md` — если уже есть, предупредить пользователя о перезаписи

**Если папки нет** — создать новую.

## Что анализировать

Обязательно изучить следующие источники:

1. **Исходный код:** связанные модули и файлы в `src/` — логика, утилиты, типы
2. **Тесты:** файлы в `test/` или рядом с исходниками — для понимания ожидаемого поведения
3. **Документация:** файлы в `docs/` — если существуют

## Структура выходного файла

Результат: `ai-process/tasks/{numberOfPlan}-{description}/analyze.md`

Анализ должен быть **на русском** и содержать следующие разделы:

```markdown
# Анализ: {название задачи}

## Общее описание функциональности
Что нужно реализовать, зачем, какую задачу решает.

## Связанные модули и сущности
Таблица: модуль/файл → назначение → что затрагивает задача.

## Текущие интерфейсы и API (если есть)
Существующие функции, классы, интерфейсы, которые затрагивает задача.

## Файлы и места в коде
Таблица: файл → что содержит → что нужно изменить/создать.

## Зависимости и ограничения
Внешние зависимости, ограничения, потенциальные риски.
```

## Уточнения
- analyze — первый этап конвейера (analyze → plan → do)
- Анализ НЕ должен содержать план реализации или примеры кода — это задача `/plan`
- Анализ должен давать достаточно контекста, чтобы на его основе можно было составить план
