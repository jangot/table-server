# Требования к сервису управления OBS/Chrome для проектора

## 1. Цель

Сервис должен автоматически поднимать и управлять локальной системой трансляции карт/вкладок браузера на проектор через OBS.

Система предназначена для сценария, в котором:

- на выделенном компьютере установлен OBS;
- OBS выводит изображение на проектор;
- Chrome используется как источник отображения карт, веб-страниц или других материалов;
- удалённый оператор отправляет ссылку, например через Telegram-бота;
- сервис на сервере принимает команду и переключает отображаемую страницу без ручного участия оператора на месте;
- после перезагрузки компьютера система должна восстановиться автоматически.

---

## 2. Общая архитектура

Сервис представляет собой локальное Node.js-приложение, работающее на компьютере, подключённом к проектору.

Сервис должен:

- запускаться автоматически после старта системы и входа пользователя в графическую сессию;
- запускать и контролировать Google Chrome;
- дожидаться готовности Chrome;
- запускать и контролировать OBS;
- принимать удалённые команды на открытие/смену URL;
- обеспечивать восстановление после падений Chrome, OBS или самого сервиса;
- иметь механизмы логирования и health-check.

---

## 3. Основные функциональные требования

### 3.1 Автозапуск

Сервис должен автоматически запускаться при старте системы.

#### Требования:
- сервис должен стартовать без ручного запуска;
- запуск должен происходить после входа пользователя в графическую сессию;
- решение должно быть совместимо с механизмом автозапуска целевой ОС;
- после перезагрузки система должна возвращаться в рабочее состояние автоматически.

---

### 3.2 Управление Chrome

Сервис должен управлять единственным экземпляром окна Chrome, используемым для трансляции.

#### Требования:
- сервис должен запускать Chrome, если он не запущен;
- Chrome должен запускаться в заранее определённом режиме;
- Chrome должен открываться в отдельном выделенном окне;
- желательно использовать полноэкранный, app или kiosk режим;
- сервис не должен создавать множество случайных окон и вкладок;
- сервис должен использовать одно постоянное окно/сессию Chrome для отображения контента;
- сервис должен уметь переключать URL в уже существующем окне.

#### Дополнительно:
- Chrome должен запускаться с включённым remote debugging port;
- сервис должен определять готовность Chrome не только по процессу, но и по доступности DevTools endpoint.

---

### 3.3 Управление OBS

Сервис должен запускать OBS только после готовности Chrome.

#### Требования:
- сервис должен запускать OBS автоматически;
- запуск OBS должен происходить только после того, как Chrome полностью поднят;
- OBS должен открываться с заранее сохранённой конфигурацией сцены;
- сцена OBS должна быть подготовлена так, чтобы сразу использовать окно Chrome как источник;
- сервис должен уметь определять, запущен ли OBS;
- при падении OBS сервис должен попытаться перезапустить его.

---

### 3.4 Переключение URL

Сервис должен принимать команду с URL и переключать текущую страницу Chrome на этот URL.

#### Требования:
- сервис должен принимать внешнюю команду на открытие ссылки;
- сервис должен валидировать входящий URL;
- после получения корректного URL сервис должен переключить страницу в уже открытом окне Chrome;
- переключение должно происходить без ручного вмешательства;
- переключение не должно требовать создания нового окна, если это не предусмотрено fallback-механизмом;
- после переключения контент должен автоматически попасть в OBS.

#### Предпочтительный способ:
- использовать Chrome DevTools Protocol (`Page.navigate`).

#### Fallback:
- если DevTools недоступен, допустим запасной способ через системные команды или скрипты автоматизации окон.

---

### 3.5 Внешний интерфейс управления

Сервис должен поддерживать внешний канал управления.

#### Минимально:
- Telegram-бот или HTTP API.

#### Требования:
- сервис должен принимать текстовую команду со ссылкой;
- сервис должен подтверждать результат выполнения команды;
- сервис должен возвращать ошибку, если ссылка некорректна или система не готова;
- сервис должен сообщать, если Chrome/OBS недоступны;
- сервис должен логировать все входящие команды.

#### Желательно:
- поддержка команд статуса;
- поддержка команды перезапуска;
- поддержка команды возврата на idle-страницу;
- поддержка whitelist доверенных пользователей.

---

## 4. Сценарии работы

### 4.1 Холодный старт после включения компьютера

Система должна выполнять следующий сценарий:

1. ОС загружается;
2. пользователь автоматически входит в систему;
3. стартует Node.js сервис;
4. сервис запускает Chrome;
5. сервис ждёт готовности Chrome;
6. сервис запускает OBS;
7. OBS открывает сохранённую сцену;
8. проектор получает сигнал;
9. система готова к приёму ссылок.

---

### 4.2 Приём новой ссылки

1. оператор отправляет ссылку;
2. сервис получает ссылку;
3. сервис валидирует ссылку;
4. сервис переключает страницу Chrome;
5. OBS продолжает захватывать окно Chrome;
6. новый контент появляется на проекторе.

---

### 4.3 Восстановление после сбоя Chrome

1. Chrome завершился аварийно;
2. сервис обнаруживает отсутствие процесса или DevTools endpoint;
3. сервис запускает Chrome заново;
4. сервис дожидается готовности Chrome;
5. при необходимости возвращает последнюю активную страницу;
6. система продолжает работу.

---

### 4.4 Восстановление после сбоя OBS

1. OBS завершился аварийно;
2. сервис обнаруживает падение;
3. сервис перезапускает OBS;
4. OBS загружает сохранённую сцену;
5. вывод на проектор восстанавливается.

---

### 4.5 Восстановление после сбоя самого сервиса

#### Требования:
- сервис должен быть запущен под процесс-менеджером;
- при аварийном завершении он должен автоматически перезапускаться;
- после перезапуска сервис должен уметь заново синхронизировать состояние системы.

---

## 5. Нефункциональные требования

### 5.1 Надёжность

Сервис должен быть устойчив к следующим ситуациям:

- перезагрузка ОС;
- падение Chrome;
- падение OBS;
- временная недоступность DevTools;
- получение некорректного URL;
- повторная отправка команды;
- запуск при уже работающих процессах.

---

### 5.2 Простота эксплуатации

Сервис должен быть удобен для эксплуатации в клубном или полукоммерческом режиме.

#### Требования:
- не должно требоваться ручное открытие приложений после обычной перезагрузки;
- оператор должен управлять системой удалённо;
- поведение системы должно быть предсказуемым;
- желательно минимизировать количество ручных действий на месте.

---

### 5.3 Производительность

#### Требования:
- переключение URL должно выполняться быстро;
- запуск системы после загрузки ОС должен занимать разумное время;
- сервис не должен существенно нагружать CPU и RAM вне моментов переключения.

---

### 5.4 Наблюдаемость

Сервис должен вести логирование.

#### Логи должны включать:
- старт сервиса;
- запуск Chrome;
- готовность Chrome;
- запуск OBS;
- входящие команды;
- успешные переключения URL;
- ошибки запуска;
- попытки перезапуска;
- состояние health-check.

#### Желательно:
- разделение логов по уровням: `info`, `warn`, `error`, `debug`.

---

## 6. Требования к состоянию и конфигурации

### 6.1 Хранимое состояние

Сервис должен хранить:

- последний успешно открытый URL;
- текущий статус Chrome;
- текущий статус OBS;
- время последнего health-check;
- список разрешённых пользователей или токенов доступа.

#### Желательно:
- восстановление последнего URL после перезапуска системы;
- возможность задать стартовую idle-страницу.

---

### 6.2 Конфигурация

Конфигурация должна быть вынесена из кода.

#### Минимальный набор параметров:
- путь к Chrome;
- путь к OBS;
- стартовый URL;
- порт DevTools;
- способ запуска;
- таймаут ожидания готовности Chrome;
- таймаут ожидания OBS;
- токен Telegram-бота или параметры HTTP API;
- список разрешённых пользователей;
- политика рестартов;
- параметры логирования.

---

## 7. Требования к безопасности

### 7.1 Валидация входящих данных

Сервис должен проверять входящие URL.

#### Требования:
- отклонять невалидные строки;
- отклонять неподдерживаемые схемы;
- при необходимости ограничить допустимые протоколы `http` и `https`;
- при необходимости ограничить домены whitelist-списком.

---

### 7.2 Ограничение доступа

Удалённое управление должно быть ограничено.

#### Требования:
- только авторизованные пользователи должны иметь возможность отправлять команды;
- команды от неизвестных пользователей должны отклоняться;
- все такие попытки должны логироваться.

---

### 7.3 Безопасный запуск внешних процессов

#### Требования:
- нельзя передавать пользовательский ввод в shell без экранирования;
- запуск Chrome и OBS должен происходить безопасным способом;
- команды должны строиться без риска command injection.

---

## 8. Требования к отказоустойчивости

Сервис должен включать watchdog-поведение.

#### Требования:
- периодическая проверка состояния Chrome;
- периодическая проверка состояния OBS;
- автоматический рестарт при сбое;
- ограничение частоты рестартов во избежание бесконечного цикла;
- фиксация причин рестарта в логах.

#### Желательно:
- отдельный статус `degraded`, если часть системы работает, а часть нет;
- отдельный статус `ready`, когда можно принимать ссылки.

---

## 9. Требования к UX/операционному использованию

Оператор должен иметь возможность:

- понять, готова ли система к работе;
- отправить ссылку;
- получить подтверждение об успехе;
- получить сообщение об ошибке;
- вручную запросить статус системы;
- вернуть систему в исходное состояние.

#### Желательные команды:
- `/status`
- `/open <url>`
- `/idle`
- `/restart chrome`
- `/restart obs`
- `/restart all`

---

## 10. Интеграционные требования

### 10.1 Интеграция с Chrome DevTools

Сервис должен уметь:
- определить активные target'ы;
- выбрать нужную вкладку/окно;
- отправить команду навигации;
- отследить успешность подключения.

---

### 10.2 Интеграция с Telegram

Если используется Telegram-бот:

#### Требования:
- сервис должен принимать команды через Bot API;
- сервис должен обрабатывать ошибки связи;
- сервис должен восстанавливаться после временной потери сети;
- сервис должен игнорировать сообщения от неавторизованных пользователей.

---

### 10.3 Интеграция с процесс-менеджером

Сервис должен корректно работать под process manager.

#### Требования:
- автоматический рестарт при падении;
- запуск при старте системы;
- сохранение логов;
- возможность ручного рестарта.

---

## 11. Ограничения и допущения

### Ограничения:
- сервис зависит от графической пользовательской сессии;
- Chrome и OBS являются GUI-приложениями и не должны запускаться как headless-only процессы;
- надёжность отображения зависит от корректной настройки OBS-сцены;
- возможны особенности ОС при работе с автологином, окнами и полноэкранным режимом.

### Допущения:
- OBS уже настроен и умеет выводить сигнал на проектор;
- сцена OBS заранее сохранена;
- проектор корректно определяется системой;
- Chrome установлен на сервере;
- Node.js окружение установлено;
- машина используется как выделенный управляющий узел.

---

## 12. Критерии готовности (Definition of Done)

Сервис считается готовым, если:

1. после перезагрузки компьютера система автоматически поднимается без ручного запуска;
2. Chrome запускается автоматически;
3. OBS запускается автоматически после готовности Chrome;
4. сервис принимает удалённую команду со ссылкой;
5. ссылка открывается в том же окне Chrome;
6. OBS продолжает корректно транслировать изображение на проектор;
7. при падении Chrome выполняется автоматическое восстановление;
8. при падении OBS выполняется автоматическое восстановление;
9. при падении Node-сервиса выполняется автоматический рестарт;
10. все ключевые события фиксируются в логах.

---

## 13. Рекомендуемые будущие улучшения

### Возможные расширения:
- web-панель управления;
- очередь команд;
- preload/preview режима перед выводом на проектор;
- белый список доменов;
- несколько сцен OBS;
- поддержка нескольких проекторов;
- расписание автоматического показа;
- локальная idle-страница с брендированным экраном ожидания;
- метрики и мониторинг;
- интеграция с NDI/WebRTC в будущем.

---

## 14. Минимальный MVP

Для первой версии достаточно реализовать:

- автозапуск Node-сервиса;
- запуск Chrome;
- ожидание готовности Chrome через DevTools;
- запуск OBS;
- Telegram-бот с командой открытия URL;
- переключение URL через DevTools;
- проверку авторизованного пользователя;
- рестарт Chrome и OBS при падении;
- логирование основных событий.
