# План: ручное управление сценами OBS

Документ описывает план реализации функций из [требований obs-scene-requirements.md](requirements/obs-scene-requirements.md): ручное управление заранее настроенными сценами OBS через Telegram-бот и Web-интерфейс.

---

## 1. Цель и границы

- **Цель:** расширить приложение возможностью **ручного** переключения сцен OBS через Telegram и Web.
- **Источник истины по сценам:** OBS. Приложение не создаёт и не конфигурирует сцены, только переключает по имени.
- **Специальные сцены:** `backup` (резервный режим), `default` (безопасное состояние) — переключение по имени и отдельные кнопки/команды.
- **Автоматическое переключение** в эту версию не входит.
- **Защита Web** (§12 требований): на данном этапе не требуется.

---

## 2. Зависимости от текущей кодовой базы

- Уже есть: backend Node.js, Telegram-бот (команды status/idle/restart, авторизация по whitelist), Express-сервер (idle + /health), модуль OBS (только **запуск процесса** OBS, без WebSocket).
- Нужно добавить: **OBS WebSocket API** — отдельный клиент/подключение для управления сценами (список, текущая, переключение). Запуск процесса OBS остаётся в существующем модуле `obs`; управление сценами — в новом сервисе/модуле, использующем WebSocket.

---

## 3. План по этапам

### Этап 1: OBS WebSocket и сервис сцен (задача 012)

- Подключение к OBS по **OBS WebSocket API** (библиотека, например `obs-websocket-js` или аналог).
- Конфигурация из env: `OBS_HOST`, `OBS_PORT`, `OBS_PASSWORD` (для WebSocket; могут отличаться от параметров запуска процесса OBS, если порт один — переиспользовать).
- Функции: установка/проверка соединения, **получение списка сцен**, **текущая сцена**, **переключение сцены по имени**.
- Обработка: OBS недоступен, разрыв WebSocket, переключение на несуществующую сцену — понятные ошибки и логи в формате key=value (например `scene_switch source=… from=… to=… success=false`).
- Логирование: подключение к OBS, восстановление соединения, переключение сцен (как в §13 требований).

**Результат:** сервис/модуль, который по запросу возвращает список сцен и текущую сцену из OBS и выполняет переключение по имени; при отсутствии конфига сцен (§5) список берётся только из OBS.

---

### Этап 2: Опциональный конфиг сцен (задача 013)

- Опциональная переменная **`SCENES_CONFIG_PATH`** — путь к JSON-файлу формата из §5 требований (name, title, type, enabled).
- Источник истины по наличию сцен — **OBS**. Конфиг используется только для отображаемых подписей (title) и типов (type) в UI. Если файла нет или имена в конфиге не совпадают со сценами в OBS — не ошибка: работа только по данным OBS.
- Сервис сцен: при наличии конфига — обогащать список сцен из OBS данными из конфига (title, type); иначе — только имена из OBS.

**Результат:** опциональное обогащение списка сцен для UI без изменения логики переключения.

---

### Этап 3: Команды Telegram для сцен (задача 014)

- Команды: **`/scenes`** (список сцен), **`/scene <name>`** (переключить), **`/current`** (текущая сцена), **`/backup`** (переключить на backup), **`/default`** (переключить на default).
- **`/status`** расширить: показывать статус OBS (connected/disconnected) и текущую сцену (пример из требований: `OBS: connected`, `Current scene: chrome`).
- Выполнение **только для whitelisted пользователей** (существующий механизм `ALLOWED_TELEGRAM_USERS`); неавторизованные — отказ и логирование.

**Результат:** полный набор Telegram-команд для управления сценами и отображения статуса.

---

### Этап 4: Web API для сцен (задача 015)

- **POST /obs/scene** — тело только JSON: `{"scene": "<name>"}`; `Content-Type: application/json`. Пример: `{"scene": "chrome"}`. Формат `application/x-www-form-urlencoded` не поддерживать.
- **POST /obs/scene/backup** — переключение на сцену `backup`.
- **POST /obs/scene/default** — переключение на сцену `default`.
- Ответы при ошибках: OBS недоступен, сцена не найдена — понятные HTTP-коды и сообщения (без реализации авторизации Web на данном этапе).

**Результат:** Web API для переключения сцен из интерфейса или внешних клиентов.

---

### Этап 5: SSR-страница управления сценами (задача 016)

- Маршрут **GET /obs/scenes**.
- Страница отображает: статус подключения к OBS, текущую сцену, список доступных сцен, кнопки переключения на каждую сцену, отдельные кнопки **Backup** и **Default**.
- Активная сцена должна быть **визуально выделена**.
- Реализация: Express + EJS (как существующая idle-страница), вызовы переключения через Web API или напрямую сервис сцен.

**Результат:** оператор может управлять сценами с одной страницы в браузере.

---

## 4. Конфигурация окружения (итог)

Добавляются/используются переменные (см. требования §14):

- `OBS_HOST` — хост OBS WebSocket (например 127.0.0.1).
- `OBS_PORT` — порт OBS WebSocket (например 4455).
- `OBS_PASSWORD` — пароль OBS WebSocket.
- `SCENES_CONFIG_PATH` — опционально, путь к JSON с подписями/типами сцен.
- `ALLOWED_TELEGRAM_USERS` — уже есть; используется для разрешения команд сцен.

---

## 5. Критерии готовности (из требований §16)

- Можно получить список сцен.
- Сцены можно переключать через Telegram.
- Сцены можно переключать через Web-интерфейс.
- Работают сцены `backup` и `default`.
- Отображается текущая сцена (в боте и на странице).
- Корректная обработка недоступности OBS (сообщения и логи).
- Неавторизованные пользователи Telegram не могут управлять системой (команды сцен под теми же правилами, что и остальные).

---

## 6. Порядок выполнения

- Этапы 1 → 2 → 3 → 4 → 5 выполняются последовательно (каждый следующий опирается на предыдущий).
- После каждого этапа — проверка работоспособности по критериям, относящимся к этому этапу.
