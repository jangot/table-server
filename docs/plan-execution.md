## 1. Анализ требований

### 1.1 Назначение системы

Сервис — локальное Node.js-приложение на машине у проектора. Оно:

- Запускает и контролирует **Chrome** (одно окно для контента).
- Запускает и контролирует **OBS** после готовности Chrome.
- Принимает **удалённые команды** (ссылка → открыть в том же окне Chrome).
- Обеспечивает **восстановление** при падении Chrome, OBS или самого сервиса.
- Запускается **автоматически** после входа пользователя в графическую сессию.

Целевой сценарий: оператор шлёт ссылку (например, через Telegram) → сервис переключает страницу в Chrome → OBS продолжает выводить окно на проектор, без действий на месте.

### 1.2 Границы и допущения (из требований)

- Зависимость от **графической сессии** — не headless.
- OBS и сцена **уже настроены**; проектор определяется системой.
- Chrome и Node.js **уже установлены** на машине.
- Машина — **выделенный узел** под эту задачу.

### 1.3 Ключевые блоки функциональности

| Блок | Содержание | Сложность |
|------|------------|-----------|
| Запуск и контроль Chrome | Отдельный модуль; запуск с remote debugging, ожидание DevTools, одно окно/режим (kiosk/app/fullscreen) | Высокая |
| Запуск и контроль OBS | Отдельный модуль; после готовности Chrome; сохранённая сцена, детекция падения | Средняя |
| Переключение URL | CDP `Page.navigate`, один target; валидация URL не в MVP | Средняя |
| Внешний интерфейс | Telegram-бот, авторизация через модуль пользователей (источник — env, затем БД), команды (open, status, idle, restart) | Средняя |
| Надёжность | Watchdog по Chrome/OBS, рестарты с ограничением частоты, восстановление последнего URL | Высокая |
| Автозапуск | Linux, PM2; инструкции по запуску при старте сессии | Средняя (больше инфра, чем кода) |
| Конфигурация и логи | Только переменные окружения; отдельный модуль валидации env; типизированные значения; уровни логов, health-check | Низкая |
| Idle-страница | **Express** + EJS-шаблоны; стартовый URL и команда `/idle` ведут на эту страницу | Низкая |
| Оркестрация запуска | Последовательный запуск внешних приложений с ожиданием готовности предыдущего; расширяемо под новые приложения | Средняя |

### 1.4 Риски и зависимости (зафиксированные решения)

- **ОС:** целевая платформа — **Linux**. От неё зависят автозапуск, пути к Chrome/OBS, документация по развёртыванию.
- **Валидация URL:** в MVP **не требуется**. При необходимости ограничения схем или доменов — после MVP.
- **Авторизация:** на первом этапе доступны только несколько пользователей; список разрешённых **логинов передаётся через переменные окружения**. Важно: с самого начала предусмотреть **отдельный модуль для работы с пользователями** (абстракция над источником списка), чтобы после MVP без смены API бота подменить источник на базу данных.
- **Process manager:** **PM2** — рестарт при падении сервиса, запуск при старте, сохранение логов. Документация и примеры конфигов под PM2.
- **Chrome и окружение:** Chrome (и при необходимости OBS) **устанавливаются вручную** до запуска сервиса. Вся система до первого запуска сервиса настраивается вручную. Требование: при старте сервиса **проверять доступность необходимого ПО** (Chrome, OBS — по путям из конфигурации, т.е. из env) и **логировать понятные ошибки**, если нужных программ в системе нет (например: «Chrome не найден по пути …», «OBS не найден …»).
- **Безопасность:** запуск Chrome/OBS без command injection; передача пользовательского ввода в аргументы процессов только с экранированием. Валидация URL — не в MVP.
- **Язык:** **TypeScript**; проект собирается/запускается из TS (сборка в JS или ts-node по выбору).
- **Idle/стартовая страница в MVP:** отдельная **idle-страница** обязательна. Реализация: **Express** как HTTP-сервер; **серверный рендеринг по EJS-шаблонам** для страницы ожидания. Chrome при старте и по команде `/idle` открывает этот URL. Порт и путь к шаблонам задаются переменными окружения.
- **Конфигурация:** вся конфигурация **строится только из переменных окружения** (без файлов конфига). **Валидация переменных окружения** выносится в **отдельный модуль конфигурации**; при старте приложения все нужные переменные проверяются, отсутствующие или невалидные приводят к понятным ошибкам и выходу. **Все значения конфигурации типизированы** (интерфейс/тип в TypeScript).
- **Внешние приложения (Chrome, OBS и др.):** в будущем количество запускаемого софта может быть расширено. **Каждое внешнее приложение реализуется в отдельном модуле** (модуль для Chrome, модуль для OBS и т.д.); логику разных приложений **не смешивать**. **Последовательный запуск:** приложения стартуют по цепочке с **ожиданием готовности предыдущего** перед запуском следующего (например: запуск Chrome → ожидание готовности DevTools → только после этого запуск OBS). Оркестрация (порядок и ожидание) должна быть явной и расширяемой под добавление новых приложений.

### 1.5 MVP по документу требований

Для первой версии достаточно:

- Проект на **TypeScript**; автозапуск Node-сервиса (через PM2).
- **Idle-страница:** локальный HTTP-сервер с рендерингом **EJS-шаблонов**; Chrome при старте и по команде `/idle` открывает этот URL.
- Запуск Chrome, ожидание готовности через DevTools.
- Запуск OBS после Chrome.
- Telegram-бот: команда открытия URL, проверка авторизованного пользователя (модуль пользователей, источник — env).
- Переключение URL через DevTools.
- Рестарт Chrome и OBS при падении.
- Логирование основных событий.

Остальное (HTTP API, расширенные команды `/idle`, `/restart`, whitelist доменов, web-панель и т.д.) — после MVP.

## 3. План выполнения (этапы)

План приведён в порядке зависимостей: от фундамента (конфигурация из env, логи, запуск процессов) к интеграциям и надёжности.

### Этап 1: Основа проекта и конфигурация

**Задача:** [001-project-foundation-and-config](../ai-process/tasks/001-project-foundation-and-config/)

- Инициализация Node-проекта на **TypeScript**: зависимости (в т.ч. типы), структура каталогов (`src/`, `src/modules/` по правилам проекта), точка входа, сборка или запуск через ts-node.
- **Конфигурация из переменных окружения:** вся конфигурация строится **только из env** (путь к Chrome, путь к OBS, порт и путь к шаблонам для idle-страницы, порт DevTools, таймауты ожидания Chrome/OBS, токен бота, список разрешённых пользователей, политика рестартов, уровень логов и т.д.). **Отдельный модуль конфигурации:** чтение env, **валидация** (наличие обязательных переменных, допустимые форматы/значения), вывод **типизированного объекта конфига** (интерфейс в TypeScript). При ошибке валидации — понятное сообщение и выход.
- Минимальное логирование: уровни (info, warn, error, debug), вывод ключевых событий (старт сервиса, запуск Chrome/OBS, команды, ошибки).
- **Idle-страница:** HTTP-сервер на **Express**; страница ожидания отдаётся через **EJS** (путь к шаблонам и порт из конфигурации). Сервер поднимается вместе с приложением; URL этой страницы — стартовый для Chrome и целевой по команде `/idle`.
- **Проверка доступности необходимого ПО при старте:** проверка наличия исполняемых файлов Chrome и OBS по путям из конфигурации; при отсутствии — понятное логирование ошибки и выход или отказ от запуска соответствующих подсистем.
- **Оркестрация запуска внешних приложений:** заложить механизм **последовательного запуска** (например, оркестратор или главный цикл): запуск первого приложения → **ожидание его готовности** (сигнал «ready» от модуля) → запуск следующего. Порядок и список приложений должны быть расширяемы (в MVP: Chrome, затем OBS). Детальная реализация модулей — на этапах 2 и 4.

**Результат:** приложение на TypeScript стартует, валидирует env через отдельный модуль, получает типизированный конфиг, поднимает idle-сервер на Express (EJS), пишет логи, при отсутствии Chrome/OBS сообщает понятные ошибки. Без запуска Chrome/OBS до реализации следующих этапов.

### Этап 2: Запуск и контроль Chrome

**Задача:** [002-chrome-launch-and-control](../ai-process/tasks/002-chrome-launch-and-control/)

- **Отдельный модуль** для работы с Chrome (логика только Chrome, без OBS и других приложений).
- Запуск Chrome с заданными аргументами (remote debugging port, режим окна — kiosk/app/fullscreen из конфигурации), без передачи пользовательского ввода в shell (без command injection).
- Ожидание готовности: проверка процесса и доступности DevTools endpoint (например, запрос к `/json/version` или подключение к порту); модуль должен предоставлять способ «дождаться старта» (Promise/колбэк «ready»), чтобы оркестратор мог запустить следующее приложение только после этого.
- Поддержка одного постоянного окна/сессии: не создавать лишние окна и вкладки.
- При необходимости: выбор нужного target по CDP и сохранение последнего открытого URL в состоянии.

**Результат:** модуль Chrome запускает процесс, дожидается готовности DevTools, экспортирует состояние «готов» для оркестрации; логика OBS в этот модуль не входит.

### Этап 3: Переключение URL через CDP

**Задача:** [003-url-switch-via-cdp](../ai-process/tasks/003-url-switch-via-cdp/)

- Подключение к DevTools (порт из конфигурации), выбор активного target (нужная вкладка/окно).
- Команда навигации: `Page.navigate` по переданному URL.
- Валидация URL в MVP не требуется; при необходимости — сохранить место под проверку схем (http/https) и whitelist доменов в следующих итерациях.
- Сохранение последнего успешного URL для восстановления после рестарта.

**Результат:** по команде (пока внутренней или тестовой) сервис переключает страницу в уже открытом окне Chrome.

---

### Этап 4: Запуск и контроль OBS

**Задача:** [004-obs-launch-and-control](../ai-process/tasks/004-obs-launch-and-control/)

- **Отдельный модуль** для работы с OBS (логика только OBS, без Chrome и других приложений).
- Запуск OBS **только после готовности Chrome**: оркестратор сначала дожидается «ready» от модуля Chrome, затем вызывает запуск OBS из модуля OBS.
- Запуск с сохранённой конфигурацией/сценой (путь из конфигурации, т.е. из env).
- Определение, запущен ли OBS (процесс и/или порт, если применимо); при необходимости модуль предоставляет сигнал готовности для последующих звеньев цепочки.
- При падении OBS — перезапуск с ограничением частоты рестартов и логированием причины (внутри модуля OBS).

**Результат:** модуль OBS запускает OBS, детектирует падение и перезапускает; порядок «сначала Chrome, потом OBS» обеспечивается оркестрацией запуска, а не смешиванием кода в одном модуле.

### Этап 5: Watchdog и восстановление Chrome/OBS

**Задача:** [005-watchdog-and-recovery](../ai-process/tasks/005-watchdog-and-recovery/)

- Периодическая проверка состояния через **соответствующие модули** (Chrome — модуль Chrome, OBS — модуль OBS), без смешивания логики.
- При падении Chrome: рестарт через модуль Chrome, ожидание готовности, при необходимости открытие последнего URL или **idle-страницы** (локальный EJS-сервер).
- При падении OBS: рестарт через модуль OBS (как в Этапе 4).
- Ограничение частоты рестартов (backoff), статусы в логах (например, `ready` / `degraded` при частичной готовности).

**Результат:** система восстанавливается после падения Chrome или OBS без бесконечных циклов рестарта; проверки и рестарты идут через отдельные модули приложений.

### Этап 6: Внешний интерфейс — Telegram-бот (MVP)

**Задача:** [006-telegram-bot-mvp](../ai-process/tasks/006-telegram-bot-mvp/)

- Приём команд через Telegram Bot API (минимум: команда с URL для открытия).
- **Модуль пользователей:** отдельный модуль для работы с разрешёнными пользователями; в MVP источник — переменные окружения (список логинов/usernames или id). Интерфейс модуля должен позволять позже подменить источник на БД без изменения кода бота.
- Проверка авторизации через этот модуль: только разрешённые пользователи могут отправлять команды.
- Вызов логики переключения URL (Этап 3) и возврат подтверждения или ошибки (система не готова, Chrome/OBS недоступны).
- Логирование всех входящих команд и отказ неавторизованным.

**Результат:** оператор может прислать ссылку боту и получить ответ; неавторизованные отклоняются.

### Этап 7: Расширенные команды и состояние (желательно в MVP или сразу после)

**Результат:** оператор может запросить статус и вернуть экран на idle; при необходимости — инициировать рестарт компонентов.

**Результат:** соответствие требованиям по безопасности и наблюдаемости; готовность к приёму как «MVP готов».
