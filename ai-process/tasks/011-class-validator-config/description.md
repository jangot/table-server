# Задача: Интеграция class-validator для валидации конфигурации

## Суть задачи

Заменить ручную валидацию переменных окружения в модуле конфигурации на использование **class-validator** (и при необходимости **class-transformer**, **reflect-metadata**). Конфигурация должна быть разбита на секции (Chrome, OBS, Telegram, idle-сервер, watchdog и т.д.); каждая секция описывается отдельным типом/интерфейсом и является полем основного класса конфига. Правила валидации задаются декораторами; сборка объекта из `process.env` и маппинг имён переменных остаются в коде, после заполнения — валидация с обработкой ошибок и выходом при невалидном конфиге.

## Детали и требования

- **Структура конфига по секциям:** конфигурация в классе разбита на секции. Конфигурация для Chrome описывается отдельным типом/интерфейсом и является полем основного конфига (например, `chrome`). Аналогично организованы конфигурации для OBS (`obs`), Telegram (`telegram`), idle-сервера, watchdog и других логических частей. Основной класс конфига содержит эти секции как поля; вложенные объекты тоже валидируются (через вложенные классы или `@ValidateNested()`).
- **Зависимости:** добавить `class-validator`; при необходимости — `class-transformer` и `reflect-metadata` (если требуются для декораторов).
- **Класс конфига и секции:** каждая секция (Chrome, OBS, Telegram и т.д.) описывается с декораторами: обязательные/опциональные строки, порты (`@IsNumber()`, `@Min(1)`, `@Max(65535)`), уровень логов, режим окна Chrome (`@IsIn([...])`), положительные целые, списки (например, разрешённые пользователи Telegram). Основной класс конфига объединяет секции и при необходимости общие поля (например, `logLevel`).
- **Маппинг env → объект:** сохранить текущую логику чтения из `process.env` и маппинг имён переменных (CHROME_PATH, IDLE_PORT и т.д.) в поля конфига; при необходимости использовать `@Transform()` из class-transformer для приведения типов (строки в числа и т.п.).
- **Обработка ошибок:** при ошибках валидации — форматировать сообщения из массива `ValidationError` (понятные пользователю), выводить в stderr и вызывать `process.exit(1)`. Желательно выводить все ошибки валидации за один раз, а не только первую.
- **Поведение и API:** сохранить текущее поведение: `getConfig()` возвращает типизированный конфиг, кэширует результат после первой успешной валидации; при ошибке — выход. Экспорт `resetConfigForTesting()` для тестов оставить. Тип конфига в сервисах — сам класс (инстанс класса используется везде как конфиг; класс служит и типом, и основой для валидации). Отдельный интерфейс `AppConfig` не нужен.
- **Тесты:** существующие тесты в `test/config.test.ts` должны проходить; при необходимости обновить их под новый способ валидации.
- **Документация:** при изменении списка переменных или формата ошибок — обновить `docs/env.md`. Описание структуры конфигурации (секции: Chrome, OBS, Telegram, idle, watchdog и т.д.) и принципа разбиения добавить в документацию в папке `docs/architecture` (новый файл или раздел в существующем).

## Контекст

- Текущая валидация: `src/modules/config/validate.ts` (ручные функции `requireEnv`, `parsePort`, `parseLogLevel`, `parseChromeWindowMode`, `parseOptionalPositiveInt` и т.д.), вход в конфиг — `src/modules/config/index.ts` (`getConfig()` вызывает `validateEnv()`).
- Типы конфига: `src/modules/config/types.ts` — сейчас интерфейс `AppConfig` и `ChromeWindowMode`; после реализации тип конфига — класс с полями-секциями (chrome, obs, telegram и т.д.), каждая секция — отдельный тип/класс.
- Загрузка env: в `src/index.ts` первым идёт `import 'dotenv/config'`; затем `getConfig()` при старте.
- Список переменных и их семантика описаны в `docs/env.md`. По итогу задачи в `docs/architecture` должно появиться описание структуры конфигурации.
