# Анализ: разделение основной и управляющих сцен OBS (main на проекторе, output для переключения)

## Общее описание функциональности

Нужно доработать интеграцию с OBS так, чтобы:

- **основная сцена `main` (или другая, заданная в конфиге)** всегда была выведена на проектор через OBS (source projector) и использовалась как базовый «холст»;
- **переключаемые пользователем сцены** (типа `output`/`input`) управляли только содержимым этого холста, но **не меняли сам факт вывода основной сцены на проектор**;
- имя сцены, которая должна выводиться на проектор, **задавалось через конфигурацию/переменные окружения**, а не вычислялось по префиксу `output.*`;
- Telegram‑бот и веб‑страница `/obs/scenes` работали **только с «управляющими» сценами**, отфильтрованными по типу/роли из конфига (например, `output`/`input`), и не могли «снять» с проектора базовую `main`.

По сути, требуется чётко развести:

- **роль сцены как проекторной (main)** — всегда висит на проекторе;
- **роль сцен как управляющих (output/input)** — доступны для переключения командами и в UI;
- **описание этих ролей и доступности** — в конфиге сцен и/или через отдельную переменную окружения для проекторной сцены.

## Текущие интерфейсы и API (если есть)

- **ObsScenesService** (`src/modules/obs-scenes/types.ts`):
  - `getScenes(): Promise<string[]>` — список имён сцен из OBS;
  - `getScenesForDisplay(): Promise<SceneForDisplay[]>` — список для UI, обогащённый данными из JSON‑конфига (`title`, `type`, `enabled`);
  - `getCurrentScene(): Promise<string | null>` — имя текущей программы OBS;
  - `setScene(name: string): Promise<void>` — переключение активной сцены по имени (без учёта ролей);
  - `disconnect()`, `isConnected()` — служебные методы.
- **SceneConfigEntry / SceneForDisplay**:
  - `SceneConfigEntry` (запись из конфига сцен): `{ name: string; title?: string; type?: string; enabled?: boolean }`;
  - `SceneForDisplay` (объект для UI): `{ name: string; title?: string; type?: string; enabled?: boolean }`;
  - поле `type` уже может использоваться для классификации (например, `working`, `backup`, `default`), но пока нигде жёстко не интерпретируется.
- **createObsScenesService** (`src/modules/obs-scenes/index.ts`):
  - принимает `ObsConfig`, `Logger` и `scenesConfigPath?`;
  - при наличии `projectorMonitorName` создаёт `onConnected`‑хук, который:
    - получает список мониторов через `client.getMonitorList()`;
    - находит монитор по имени `projectorMonitorName` (или префиксу);
    - получает список сцен `client.getSceneList()`;
    - выбирает **первую сцену с именем, начинающимся на `output.`**;
    - вызывает `client.openSourceProjector(outputScene.sceneName, monitor.monitorIndex)`.
  - загружает конфиг сцен через `loadScenesConfigSync(scenesConfigPath)` и передаёт в `createObsScenesServiceImpl`.
- **Idle‑сервер и web‑UI**:
  - `GET /obs/scenes` (`src/modules/idle-server/index.ts`):
    - берёт `connected = obsScenes.isConnected()`;
    - параллельно получает `currentScene` и `scenes = obsScenes.getScenesForDisplay()`;
    - рендерит `views/obs-scenes.ejs` с этими данными.
  - `views/obs-scenes.ejs`:
    - показывает статус подключения к OBS и текущую сцену;
    - для списка кнопок использует `scenes.filter(s => s.enabled !== false)` — все сцены, кроме явно отключённых;
    - каждая кнопка вызывает `switchScene(name)` → `POST /obs/scene` (или спец‑маршруты для `backup`/`default`).
- **Telegram‑бот** (`src/modules/telegram-bot/handlers.ts`):
  - `/scenes` (`handleScenes`):
    - берёт `scenes = deps.obsScenes.getScenesForDisplay()`;
    - выводит пользователю список `title ?? name` **без дополнительной фильтрации по типу**;
  - `/scene <name>` (`handleScene`):
    - напрямую вызывает `deps.obsScenes.setScene(sceneName)` по имени из команды, без проверки роли;
  - `/backup` и `/default`:
    - используют `switchToNamedScene('backup'/'default', ...)`, который вызывает `setScene` с жёстко зашитыми именами;
  - `/current`, `/status`:
    - используют `getCurrentScene()` и не завязаны на разделение `main`/`output`.
- **Конфиг и окружение**:
  - `ObsConfig` (`src/modules/config/types.ts`):
    - уже содержит `projectorMonitorName?: string` — имя монитора для вывода проектора;
    - **нет поля для имени сцены, выводимой на проектор**;
  - `validateEnv()` (`src/modules/config/validate.ts`):
    - читает `SCENES_CONFIG_PATH` в `scenesConfigPath` и OBS‑переменные (`OBS_PATH`, `OBS_HOST`, `OBS_PORT`, `OBS_PASSWORD`, `OBS_PROJECTOR_MONITOR_NAME` и др.);
    - не читает имя проекторной сцены (его пока нет в требованиях/конфиге).

Сейчас **нет явной договорённости**, какая сцена считается «основной» для проектора, а какая — управляющей. Фактически для проектора используется любая сцена с префиксом `output.`, а все сцены (кроме отключённых через `enabled: false`) доступны для переключения в UI и через Telegram.

## Зависимости и ограничения

- **Источник истины по сценам по‑прежнему OBS.** Список сцен и факт существования сцены берутся из OBS WebSocket API (`getSceneList`, `setCurrentProgramScene`). JSON‑конфиг сцен остаётся вспомогательным для UI (подписи, типы, флаги, роли).
- **Существующий JSON‑конфиг сцен должен остаться обратно совместимым.** Уже реализованный формат (`name`, `title`, `type`, `enabled`) и поведение `loadScenesConfigSync` не должны ломаться. Введение ролей `main`/`output`/`input` должно либо опираться на существующее поле `type`, либо аккуратно его расширять, чтобы старые конфиги в стиле задач 012/013 продолжали работать.
- **Проекторная сцена не должна «слетать» из‑за пользовательских действий.** Логика Telegram‑бота и веб‑интерфейса должна гарантировать, что:
  - команды `/scene`, `/backup`, `/default` и кнопки на странице не могут переключить сцену с проекторной `main` на другую, если по архитектуре требуется всегда держать `main` на проекторе;
  - переключения касаются только управляющих сцен (`output`/`input`), а `main` остаётся основой для вывода.
- **Переменная окружения для проекторной сцены должна быть опциональной, но однозначной.** При её отсутствии нужно определить поведение по умолчанию (например, использовать старый механизм `output.*` или вообще не запускать projector и только логировать предупреждение). Это поведение следует явно описать и задокументировать.
- **Требования безопасности и авторизации сохраняются.** Task 030 не меняет модель доступа: web‑интерфейс и Telegram‑бот остаются защищёнными так же, как сейчас; новые возможности (выбор ролей сцен) не должны открывать дополнительные пути неавторизованного управления OBS.
- **Ошибки/edge‑кейсы должны обрабатываться мягко.** Если в OBS нет сцены с именем из переменной окружения, если роли сцен в конфиге заданы некорректно или если конфиг отсутствует — приложение не должно падать, а должно логировать понятные сообщения и переходить к безопасному состоянию (например, без запуска проектора или с минимальной функциональностью управления сценами).
