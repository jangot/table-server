# Задача: Разделить основную и управляющие сцены OBS (main на проекторе, output для переключения)

## Суть задачи
В текущей конфигурации OBS сцена для проекторного вывода и сцены, которыми управляют команды (веб-страница `/obs/scenes` и Telegram‑бот), не разделены по ролям. Нужно реализовать схему, при которой на проектор (монитор) всегда выводится одна основная сцена `main` (проходит через фильтры и используется как базовый «холст»), а команды переключают только сцены типа `output`/`input`, не влияя на сам факт вывода `main` на проектор. Имя основной сцены должно задаваться через переменные окружения. Команды Telegram и веб‑интерфейс должны работать только с «управляющими» сценами (например, `output.*` или сцены, помеченные в конфиге), а не с базовой `main`.

## Детали и требования
- **Структура сцен в OBS (как описано пользователем):**
  - `main` — основная сцена, которая выводится на проектор и проходит через нужные фильтры; именно её нужно держать на проекторе постоянно.
  - `output` — сцена‑«сборка» всех доступных сцен вместе (агрегирующая/композитная сцена).
  - `input`‑сцены — отдельные составные сцены с конкретными визуализациями.
- **Требование к проектору:**
  - На проектор (через OBS projector/source projector) должна выводиться всегда сцена `main`, а не автоматически найденная по префиксу `output.` (как сейчас в коде).
  - Имя сцены, выводимой на проектор, должно задаваться через переменную окружения (например, `OBS_PROJECTOR_SCENE_NAME`, точное имя будет определено на этапе анализа/планирования).
  - Текущая логика в `src/modules/obs-scenes/index.ts`, где для проектора находится первая сцена с `sceneName.startsWith('output.')`, должна быть переработана с учётом нового способа выбора сцены.
- **Требование к командам и переключению сцен:**
  - Команды Telegram (`/scene`, `/backup`, `/default`, `/scenes`) и веб‑страница `/obs/scenes` должны управлять только «управляющими» сценами (output/input), но не трогать базовую `main` на проекторе.
  - Список сцен, доступных для переключения, формируется через `ObsScenesService.getScenesForDisplay()` и используется:
    - в HTTP‑слое idle‑сервера (`/obs/scenes`, шаблон `views/obs-scenes.ejs`);
    - в Telegram‑боте (`handleScenes`, `handleScene`, `handleBackup`, `handleDefault` в `src/modules/telegram-bot/handlers.ts`).
  - Нужно продумать, как по конфигурации (например, через `scenesConfigPath` и JSON‑конфиг сцен) выделить:
    - сцену, которая всегда должна висеть на проекторе (тип `main` или отдельное поле);
    - сцены, допустимые для переключения пользователем (типы `output`, `input` или флаг `enabled`).
- **Переменные окружения и конфиг:**
  - В `.env` уже используется блок OBS‑настроек через `ObsConfig` (`OBS_PATH`, `OBS_HOST`, `OBS_PORT`, `OBS_PASSWORD`, `OBS_PROJECTOR_MONITOR_NAME` и т.д.).
  - Нужно добавить/уточнить переменную окружения для имени сцены, выводимой на проектор (например, `OBS_PROJECTOR_SCENE_NAME=main`), и связать её с кодом `createObsScenesService`.
  - Опционально — расширить JSON‑конфиг сцен (`SCENES_CONFIG_PATH` → `scenesConfigPath` → `loadScenesConfigSync`) полем `type`/`role` (`main` / `output` / `input`) и/или дополнительными флагами, чтобы:
    - UI и Telegram показывали только нужные сцены;
    - внутренняя логика могла отличать `main` от остальных.
- **Изменяемые модули (ожидаемо):**
  - `src/modules/obs-scenes/index.ts` — логика выбора монитора и сцены для `openSourceProjector`.
  - `src/modules/obs-scenes/scenes-service.ts` — построение списка `SceneForDisplay` с учётом ролей/типов сцен и их доступности для переключения.
  - `src/modules/idle-server/index.ts` и `views/obs-scenes.ejs` — отображение и переключение сцен в веб‑интерфейсе.
  - `src/modules/telegram-bot/handlers.ts` — команды `/scenes`, `/scene`, `/backup`, `/default` должны работать только с нужной группой сцен.
  - Конфиг и переменные окружения: `.env`, `src/modules/config/validate.ts`, `src/modules/config/types.ts` (добавление поля для имени проекторной сцены).

## Контекст
- Проект уже использует модуль `obs-scenes` и OBS WebSocket (`src/modules/obs-scenes/client.ts`, `scenes-service.ts`, `index.ts`) для:
  - получения списка сцен (`getSceneList`, `getScenes`, `getScenesForDisplay`);
  - получения текущей программы (`getCurrentProgramScene`, `getCurrentScene`);
  - переключения сцены (`setCurrentProgramScene`, `setScene`);
  - открытия source projector на указанный монитор (`openSourceProjector`, `getMonitorList`).
- Выбор монитора для проектора сейчас делается по имени из `OBS_PROJECTOR_MONITOR_NAME` (через `ObsConfig.projectorMonitorName`), а сцена для проектора выбирается как первая сцена с именем, начинающимся на `output.`.
- Список сцен и их отображение:
  - Веб‑страница `/obs/scenes` (`idle-server` + `views/obs-scenes.ejs`) получает `scenes` через `obsScenes.getScenesForDisplay()` и фильтрует по `enabled !== false`.
  - Telegram‑бот (модуль `telegram-bot`) также использует `getScenesForDisplay()` для списка сцен (`/scenes`) и `setScene()` для переключения (`/scene <name>`, `/backup`, `/default`).
- Конфигурация сцен уже поддерживается через отдельный JSON‑файл (`SCENES_CONFIG_PATH`), который парсится в `SceneConfigEntry` (поля `name`, `title`, `type`, `enabled`), и может быть расширен для описания ролей `main`/`output`/`input`.

## Открытые вопросы
- Как именно классифицировать сцены в конфиге:
  - использовать ли существующее поле `type` (например, значения `main`, `output`, `input`);
  - или завести отдельное поле/флаг для «проекторной» сцены и для сцен, доступных для переключения.
- Должны ли Telegram‑команды `/backup` и `/default` жёстко привязываться к сценам с именами `backup`/`default`, или их тоже нужно описывать через конфиг (например, как алиасы к определённым `input`/`output`‑сценам).
- Нужно ли отображать сцену `main` в списке `/scenes` и на веб‑странице `/obs/scenes`, если она всегда закреплена за проектором и не должна переключаться командами.
- Требуется ли дополнительный fallback‑сценарий, если сцена, указанная в переменной окружения для проектора, не найдена в OBS (например, логирование и отсутствие запуска проектора, либо попытка использовать старую логику `output.*` как резервный вариант).
