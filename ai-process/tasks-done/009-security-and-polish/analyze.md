# Анализ: Безопасность и полировка (задача 009)

## Общее описание функциональности

Задача — проверить защиту от command injection при запуске Chrome/OBS и при обработке пользовательского ввода, заложить или уточнить подход к валидации URL (вне MVP), обеспечить логирование отклонённых запросов при будущей валидации. Кроме того — сверить реализацию с критериями готовности (DoD) из раздела 12 требований и при необходимости дополнить логи, чтобы ключевые события были зафиксированы.

Итог: соответствие требованиям по безопасности и наблюдаемости; готовность к приёму как «MVP готов».

## Текущие интерфейсы и API (если есть)

- **Chrome:** `launchChrome(chromePath, args, port, timeoutMs, logger)` — аргументы формируются в `buildChromeArgs(config, devToolsPort, initialUrl)`, где `initialUrl` всегда `http://localhost:${config.idlePort}/`. Пользовательский URL в аргументы процесса не передаётся.
- **OBS:** `launchObs(obsPath, args, timeoutMs, logger)` — аргументы из `buildObsArgs(config)` (только config, в т.ч. `obsProfilePath`).
- **Навигация:** `navigateToUrl(url, deps)` (chrome) → `cdpNavigateToUrl(port, url, statePath, logger)` — URL передаётся в Puppeteer `page.goto(url)` и в `writeLastUrl(statePath, url)`. В shell или в аргументы `spawn` не попадает.
- **Конфиг:** `validateEnv()` возвращает `AppConfig`; все поля (включая `chromePath`, `obsPath`, пути, порты) заполняются только из переменных окружения.

## Зависимости и ограничения

- **Безопасность:** В коде не используется `shell: true`, `exec` или `execSync`. Запуск Chrome и OBS — только `spawn(path, args, { shell: false })`, пути и аргументы берутся из конфига (env). Пользовательский URL используется только в CDP (`page.goto`) и при записи в файл последнего URL — в shell не подставляется. Команда `/restart` использует пользовательский ввод только для выбора одного из трёх фиксированных значений (`chrome`/`obs`/`all`), сами вызовы `restartChrome`/`restartObs` получают только config и logger. Имеет смысл явно зафиксировать это в коде (комментарии или вынос в отдельный короткий документ/чек-лист), чтобы при дальнейших изменениях не ослабить защиту.
- **Валидация URL:** По плану (1.4, этап 9) в MVP не входит. При добавлении позже — проверка схем (http/https), при необходимости whitelist доменов, и обязательное логирование отклонённых запросов. Сейчас в боте извлекается подстрока по regex `https?:\/\/\S+` без проверки схемы/домена; место для будущей валидации — до вызова `navigateToUrl` в handlers и/или на входе в `navigateToUrl`/cdp.
- **DoD (раздел 12 требований):** Пункт 10 — «все ключевые события фиксируются в логах». Остальные пункты (1–9) описывают поведение системы (автозапуск, приём команды, открытие ссылки, восстановление и т.д.); логи должны позволять убедиться, что эти события происходят. Текущее покрытие:
  - Старт сервиса, старт модулей (Chrome, OBS), готовность DevTools/OBS — есть.
  - Приём команды/URL из Telegram — есть (navigated to URL, unauthorized, system not ready, navigate failed).
  - Watchdog: ready/degraded, ошибки рестарта Chrome/OBS — есть.
  - OBS: выход процесса, лимит рестартов, ошибка рестарта — есть.
  - Нет отдельного лога при успешном открытии ссылки «в том же окне» после команды — это по смыслу покрыто «Navigated to URL» и «Telegram bot: navigated to URL». При необходимости можно уточнить формулировки или добавить один общий лог «remote command processed» с типом (open/idle/restart/status).
- **Риски:** Любое будущее изменение, добавляющее вызов `exec`/`execSync` или `spawn(..., { shell: true })` с подстановкой пользовательского ввода, создаёт риск command injection — такие места должны проходить явную проверку. Добавление валидации URL без логирования отклонённых запросов снизит наблюдаемость.

## Краткое резюме для плана

1. **Command injection:** Пройти все места запуска процессов (Chrome, OBS) и все использования пользовательского ввода (Telegram: URL, текст команды `/restart`) — подтвердить отсутствие передачи в shell; при необходимости добавить комментарии в код или короткий чек-лист в документации.
2. **Валидация URL:** В MVP не реализовывать; оставить в описании/плане требование: при появлении валидации — схемы (http/https), при необходимости whitelist, и логирование отклонённых запросов.
3. **DoD и логи:** Сверить пункты 1–10 раздела 12 с текущими логами; при необходимости добавить или уточнить сообщения так, чтобы ключевые события (старт, готовность, приём команды, навигация, рестарты, отказы) были однозначно видны в логах.
