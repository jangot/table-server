# Анализ: Проанализировать проект и сделать README.md

## Общее описание функциональности

Нужно подготовить полноценный **README.md** в корне репозитория. README должен давать новому разработчику или оператору понятное описание: что это за сервис, зачем он нужен, как его установить, настроить и запустить.

Задача решает отсутствие корневого README (файла сейчас нет) и пустое поле `description` в `package.json`. Целевая аудитория — разработчики и операторы, которые клонируют репозиторий и хотят быстро понять назначение сервиса, требования к окружению и порядок запуска.

Фактическое назначение сервиса (по коду и `docs/requirements/init.md`):

- **Управление OBS и Chrome** для трансляции на проектор: сервис запускает Chrome и OBS, ждёт их готовности, переключает URL в Chrome через CDP.
- **Приём команд** через Telegram-бота (открытие URL, статус, перезапуск chrome/obs, переключение сцен OBS) и через HTTP (idle-сервер: health, переключение сцен OBS).
- **Автозапуск и восстановление**: watchdog перезапускает Chrome/OBS при падении; развёртывание под PM2 с автозапуском после входа в графическую сессию описано в `docs/deployment-pm2.md`.

Конфигурация — только через переменные окружения (и опционально `.env`); валидация при старте, при ошибке — выход с кодом 1 и списком нарушений.

**Язык README:** в описании задачи не зафиксирован. По правилам проекта landing page и подобный контент — на английском; для внутренней документации допустим русский. Для открытого репозитория и унификации с типичной практикой (GitHub/GitLab) целесообразно сделать README **на английском**; при необходимости можно добавить краткую секцию на русском или оставить только английский.

## Текущие интерфейсы и API (если есть)

- **Конфиг:** `getConfig()` из `src/modules/config` возвращает валидированный `AppConfig`; при ошибке валидации — вывод в stderr и `process.exit(1)`.
- **Idle HTTP API:** `GET /` — главная (idle), `GET /health` — JSON `{ ready, chrome, obs }`, `GET /obs/scenes` — страница со сценами, `POST /obs/scene` с телом `{ scene: string }`, `POST /obs/scene/backup`, `POST /obs/scene/default`.
- **Telegram-бот:** команды перечислены в `handleHelp` в `handlers.ts` (/status, /idle, /restart, /scenes, /scene, /current, /backup, /default, /help; текст с URL открывает страницу в Chrome). Доступ только для пользователей из `ALLOWED_TELEGRAM_USERS`.
- **Скрипты:** `npm run build` (tsc), `npm start` (build + node dist/index.js), `npm run dev` (ts-node src/index.ts), `npm test` (node test runner для перечня тестов), `npm run lint` (eslint).

Для README достаточно описать назначение сервиса, требования (Node.js, Chrome, OBS, графическая сессия), установку зависимостей, настройку (env / .env.example), запуск и при необходимости кратко — команды бота и HTTP API; детали API уже есть в docs.

## Зависимости и ограничения

- **Внешние зависимости:** Node.js (LTS рекомендуется), Chrome/Chromium, OBS Studio — должны быть установлены и доступны по путям из конфига; сервис проверяет исполняемые файлы при старте. Графическая сессия обязательна (Chrome и OBS — GUI). Документация ссылается на PM2 для развёртывания и автозапуска.
- **Ограничения:** README не должен подменять собой детальную документацию в `docs/` (env.md, deployment-pm2.md, requirements) — достаточно кратких разделов и ссылок. Не включать в README секреты и реальные значения из .env.
- **Риски:** Если описать только часть переменных в README, возможна рассинхронизация с docs/env.md и validate.ts — предпочтительно дать в README минимальный набор (обязательные + пара опциональных) и ссылку на docs/env.md для полного списка. Правила проекта: landing page и подобный контент на английском — для README в корне репозитория разумно принять английский как основной язык.
