# Анализ: Основа проекта и конфигурация

## Общее описание функциональности

Задача — инициализировать Node.js-проект на TypeScript с модульной структурой и заложить фундамент для последующих этапов (запуск Chrome, OBS, Telegram-бот). Реализуется только инфраструктура и конфигурация; сами модули Chrome и OBS в этой задаче не реализуются.

Нужно:

1. **Структура проекта:** каталоги `src/`, `src/modules/`, точка входа, сборка или запуск через ts-node (или компиляция в JS).
2. **Конфигурация из env:** один источник — переменные окружения. Отдельный модуль конфигурации: чтение env, валидация (обязательные переменные, допустимые форматы), типизированный объект конфига. При ошибке валидации — понятное сообщение и выход.
3. **Логирование:** уровни (info, warn, error, debug), вывод ключевых событий (старт сервиса, запуск приложений, команды, ошибки).
4. **Idle-страница:** HTTP-сервер на Express, рендеринг страницы ожидания через EJS (порт и путь к шаблонам из конфига). URL этой страницы — стартовый для Chrome и целевой по команде `/idle`.
5. **Проверка ПО при старте:** проверка наличия исполняемых Chrome и OBS по путям из конфига; при отсутствии — понятное логирование и выход или отказ от запуска соответствующих подсистем.
6. **Оркестрация:** механизм последовательного запуска приложений: запуск первого → ожидание сигнала «ready» от модуля → запуск следующего. Порядок расширяем (в MVP: Chrome, затем OBS). Реализацию модулей Chrome/OBS — на этапах 2 и 4, здесь только каркас оркестратора.

Результат этапа: приложение стартует, валидирует env, поднимает idle-сервер (Express/EJS), пишет логи, при отсутствии Chrome/OBS сообщает понятные ошибки. Без реального запуска Chrome/OBS.

## Текущие интерфейсы и API (если есть)

Интерфейсов и API в кодовой базе нет. Проект инициализирован минимально (`package.json` без TypeScript и без зависимостей Express/EJS). Требуется определить:

- Интерфейс конфига (TypeScript): поля для путей к Chrome/OBS, портов (idle, DevTools), путей к шаблонам EJS, таймаутов, токена бота, списка пользователей, политики рестартов, уровня логов и т.д.
- Контракт модуля приложения для оркестратора: способ запуска и способ сигнализировать «ready» (например, функция возвращающая `Promise<void>` или колбэк).
- Публичный API модуля конфигурации: одна функция/объект, возвращающий валидированный конфиг или выбрасывающий при ошибке.

## Зависимости и ограничения

- **Целевая ОС:** Linux. Проверка исполняемых и пути — с учётом Linux (исполняемый файл по пути из env).
- **Конфигурация:** только env, без файлов конфига. Имена переменных окружения нужно зафиксировать в коде и/или в документации.
- **Зависимости:** Node.js, TypeScript, Express, EJS; опционально — библиотека для логирования с уровнями (например, pino, winston) или минимальная своя обёртка.
- **Риски:** при неверном формате портов/путей валидация должна сообщать понятную ошибку и завершать процесс; не должно быть «тихого» падения.
- **Связь с последующими этапами:** модули Chrome (002) и OBS (004) должны интегрироваться в оркестратор и конфиг; интерфейс «запуск + ready» нужно заложить уже на этапе 1.
- **Без реализации в задаче 001:** реальный запуск процессов Chrome и OBS, CDP, Telegram-бот, модуль пользователей — только конфиг и оркестрация под них.
