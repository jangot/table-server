# Анализ: Использование dotenv для переменных окружения

## Общее описание функциональности

Нужно перейти с ручного задания переменных окружения через `export` в shell на загрузку из файла `.env` при старте приложения с помощью библиотеки **dotenv**. Цель — упростить локальную разработку и запуск: разработчик создаёт `.env` один раз и не выполняет экспорт вручную. Текущий способ с `export` остаётся опциональным: переменные, уже заданные в окружении процесса, имеют приоритет над значениями из файла (стандартное поведение dotenv).

## Связанные модули и сущности

| Модуль/файл | Назначение | Что затрагивает задача |
|-------------|------------|------------------------|
| `src/index.ts` | Точка входа: вызов `getConfig()`, затем инициализация логгера, проверки, серверы | Загрузка dotenv должна выполняться **до** первого обращения к конфигу (до любого чтения `process.env`) |
| `src/modules/config/` | Чтение и валидация переменных из `process.env`, кэшированный типизированный конфиг | Не меняет логику валидации; конфиг по-прежнему читает `process.env`, но к моменту вызова `getConfig()` переменные уже могут быть заполнены из `.env` |
| `docs/env.md` | Документация переменных окружения и пример через `export` | Нужно описать использование `.env` и наличие `.env.example` |
| `.gitignore` | Исключение файлов из репозитория | Проверить наличие `.env` (уже есть) |
| `package.json` | Зависимости и скрипты запуска | Добавить зависимость `dotenv`; при необходимости упомянуть порядок загрузки в скриптах |

## Текущие интерфейсы и API (если есть)

- **`getConfig(): AppConfig`** (`src/modules/config/index.ts`) — при первом вызове вызывает `validateEnv()`, который читает `process.env` через внутреннюю `getEnv(name)`. Кэш сбрасывается в тестах через `resetConfigForTesting()`.
- **`validateEnv(): AppConfig`** (`src/modules/config/validate.ts`) — читает обязательные и опциональные переменные (`CHROME_PATH`, `OBS_PATH`, `IDLE_PORT`, `IDLE_VIEWS_PATH`, `LOG_LEVEL`, опционально `DEVTOOLS_PORT`, `CHROME_READY_TIMEOUT`, `OBS_READY_TIMEOUT`, `CHROME_WINDOW_MODE`), валидирует форматы, при ошибке бросает `Error`. API менять не требуется.
- Точка входа не экспортирует конфиг наружу; единственное место, где вызывается `getConfig()` — `src/index.ts` в `main()`.

## Файлы и места в коде

| Файл | Что содержит | Что нужно изменить/создать |
|------|----------------|----------------------------|
| `package.json` | Зависимости (dotenv отсутствует), скрипты `start`, `dev` | **Добавить** зависимость `dotenv`. При необходимости — уточнить, загружать ли dotenv через код в `src/index.ts` или через флаг Node (например `node -r dotenv/config`). |
| `src/index.ts` | Импорты, `main()` с вызовом `getConfig()` первым делом | **Изменить**: подключить и вызвать загрузку dotenv (например `import 'dotenv/config'` или `require('dotenv').config()`) в самом начале файла, до вызова `getConfig()`, чтобы переменные из `.env` попали в `process.env` до чтения конфига. |
| `.gitignore` | Уже есть строка `.env` | **Проверить** — при наличии ничего не менять; при отсутствии — **добавить** `.env`. |
| `.env` | Не в репозитории (игнорируется) | **Не коммитить**. Используется локально каждым разработчиком. |
| `.env.example` | Сейчас отсутствует | **Создать**: пример перечня переменных (без секретов) и краткого формата для копирования в `.env`. |
| `docs/env.md` | Описание переменных, пример через `export`, предупреждение не коммитить секреты и использовать `.env` локально + `.gitignore` | **Дополнить**: явно описать, что приложение при старте загружает переменные из `.env` (если файл есть); указать на `.env.example` как образец; при желании оставить пример с `export` как альтернативу. |

## Зависимости и ограничения

- **Зависимость:** добавить пакет `dotenv` (обычно в `dependencies`, т.к. используется в рантайме при старте приложения).
- **Порядок инициализации:** загрузка dotenv должна произойти до любого кода, который читает `process.env`. Сейчас таким кодом является вызов `getConfig()` в `main()`. Импорты в `index.ts` поднимаются до выполнения `main()`, но сами модули (`config`, `logger` и т.д.) не читают `process.env` на уровне модуля — только при вызове `getConfig()`. Поэтому достаточно вызвать `dotenv.config()` (или подключить `dotenv/config`) в начале `src/index.ts` до вызова `getConfig()`.
- **Тесты:** в `test/config.test.ts` переменные задаются через `process.env` в `before()` и очищаются в `after()`. Тесты не проходят через `src/index.ts`, а импортируют `getConfig`/`validateEnv` напрямую. Загрузка dotenv в точке входа на тесты не влияет; при необходимости тесты могут по-прежнему переопределять переменные. Риск: если в тестовом окружении есть файл `.env` в cwd и тесты не сбрасывают переменные, значения из `.env` могут попасть в тесты — текущая схема с `before`/`after` и `resetConfigForTesting()` перезаписывает нужные переменные и чистит опциональные, так что поведение должно остаться предсказуемым.
- **Окружения (dev/stage/prod):** в задаче не требуется разный набор переменных по окружениям; достаточно одного `.env`/`.env.example`. При необходимости разные окружения могут использовать разные файлы (например `.env.production`) или задавать переменные в process manager (PM2 и т.п.) без изменения кода текущей задачи.
