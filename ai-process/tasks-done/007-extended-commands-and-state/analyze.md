# Анализ: Расширенные команды и состояние (007)

## Общее описание функциональности

Нужно добавить расширенные команды Telegram-бота и механизм проверки готовности системы:

- **Команды бота:** `/status` (вывод статуса системы и Chrome/OBS), `/idle` (переключение на URL idle-страницы — локальный Express/EJS-сервер). При необходимости — `/restart chrome`, `/restart obs`, `/restart all` для ручного инициирования рестарта компонентов.
- **Health-check:** способ узнать готовность системы и статус Chrome/OBS — либо отдельный HTTP endpoint, либо ответ в боте (по заданию допустимы оба варианта).
- **Сохранение и восстановление последнего URL при рестарте сервиса:** по документу плана это могло быть сделано на Этапе 2/3 — в коде уже реализовано (модуль `chrome/lastUrlState.ts`, конфиг `lastUrlStatePath`, использование при `restartChrome` и при каждой навигации). Задача — убедиться, что это покрыто и при необходимости доработать.

Результат: оператор может запросить статус, вернуть экран на idle по команде `/idle`, при необходимости инициировать рестарт Chrome, OBS или обоих.

## Текущие интерфейсы и API (если есть)

- **TelegramBotDeps** (`telegram-bot/types.ts`): `config`, `logger`, `allowedUsers`, `navigateToUrl(url, deps)`, `isChromeAlive(config)`, `isObsAlive(config)`. Обработки команд по типу `/status` или `/idle` нет; бот обрабатывает только `bot.on('text')` и трактует сообщение как URL, если из текста извлекается ссылка.
- **Chrome:** `navigateToUrl(url, { config, logger })`, `restartChrome(config, logger)`, `isChromeAlive(config)`. Idle URL в коде: `http://localhost:${config.idlePort}/`.
- **OBS:** `isObsAlive()`, `restartObs(config, logger)` (сигнатура для единообразия с watchdog; внутри не использует config/logger для логики рестарта).
- **lastUrlState:** `readLastUrl(filePath)`, `writeLastUrl(filePath, url)` — уже используются при навигации и при рестарте Chrome; восстановление последнего URL при рестарте сервиса реализовано.
- Отдельного HTTP endpoint для health-check в проекте нет; watchdog ведёт внутренний статус `ready`/`degraded` только в логах.

| Альтернатива: отдельный модуль/файл для health | — | Если health выносить в отдельный модуль — определить место (например `src/modules/health/` или маршрут в idle-server) и формат ответа |

Уточнение по восстановлению URL: в `chrome/launch.ts` при старте Chrome открывается `idleUrl`; после оркестратора восстановление последнего URL не вызывается при первом старте приложения (только при рестарте Chrome из watchdog или вручную). В `restartChrome` уже читается `readLastUrl` и выполняется переход на lastUrl или idle. При полном рестарте сервиса (перезапуск процесса) Chrome поднимается заново и открывает только idleUrl; восстановление lastUrl при старте процесса в коде не делается — это может быть отдельным решением (после готовности Chrome вызвать навигацию на lastUrl из файла), если это входит в объём задачи.

## Зависимости и ограничения

- **Зависимости задачи:** Этап 6 (Telegram-бот MVP), переключение URL (Этап 3), idle-страница (Этап 1), watchdog (Этап 5) — всё уже есть в коде.
- **Авторизация:** команды `/status`, `/idle`, `/restart` должны быть доступны только разрешённым пользователям (тот же `allowedUsers`, что и для текста с URL).
- **Telegraf:** для команд используется `bot.command('status', handler)` и т.д.; при наличии и текста, и команд нужно либо регистрировать команды до общего `on('text')`, либо в обработчике `text` игнорировать сообщения, начинающиеся с `/`, чтобы не трактовать их как URL.
- **Риски:** ручной рестарт Chrome/OBS может конфликтовать с watchdog по частоте рестартов (существующие ограничения в watchdog и в OBS-модуле остаются); при добавлении HTTP health на idle-server — не раздувать обязанности idle-server (один маршрут `/health` с минимальной логикой допустим).
- **Внешние зависимости:** только уже используемые (Telegraf, конфиг, модули chrome/obs). Отдельный HTTP-сервер для health не обязателен, если решено ограничиться ответом в боте по `/status`.
