# Анализ: Watchdog и восстановление Chrome/OBS

## Общее описание функциональности

Нужно реализовать **watchdog** — периодическую проверку состояния Chrome и OBS через соответствующие модули. При обнаружении падения выполнять рестарт (Chrome — через модуль Chrome, OBS — через модуль OBS) с ограничением частоты рестартов (backoff). Для Chrome после восстановления открывать последний URL или idle-страницу.

**Уточнение:** интервал периодической проверки статуса (как часто опрашивать Chrome и OBS) должен задаваться через **переменные окружения**, а не быть зашит в коде.

Задача решает надёжность системы: после падения браузера или OBS сервис восстанавливает работу без ручного вмешательства и без бесконечных циклов рестарта за счёт backoff и явных статусов в логах (ready / degraded).

## Связанные модули и сущности

| Модуль/файл | Назначение | Что затрагивает задача |
|-------------|------------|-------------------------|
| `src/modules/chrome` | Запуск Chrome, ожидание DevTools, навигация, последний URL | Проверка «жив ли Chrome»; рестарт; открытие last URL или idle после рестарта |
| `src/modules/obs` | Запуск OBS, ожидание готовности, рестарт по событию `exit` | Проверка «жив ли OBS»; возможность рестарта по инициативе watchdog (вне события exit) |
| `src/modules/config` | Валидация env, типы конфига | Новые переменные: интервал проверки статуса (и при необходимости backoff/политика рестартов) |
| `src/modules/orchestrator` | Последовательный запуск модулей (Chrome → OBS) | Запуск watchdog после старта цепочки (или интеграция цикла проверок) |
| `src/index.ts` | Точка входа, создание модулей, запуск оркестратора | Подключение watchdog (новый модуль или цикл после runOrchestrator) |
| `src/modules/logger` | Логирование | Статусы ready/degraded и события рестартов |
| `docs/env.md` | Документация переменных окружения | Описание новых переменных для интервала проверки |

## Текущие интерфейсы и API (если есть)

- **Chrome**
  - `createChromeModule(config, logger)` — возвращает `AppModule` с `start()` (один запуск при старте).
  - `getChromeProcess(): ChildProcess | null` — текущий процесс Chrome.
  - `launchChrome(chromePath, args, port, timeoutMs, logger)` — запуск и ожидание DevTools.
  - `waitForDevTools(port, timeoutMs)` — опрос `http://127.0.0.1:port/json/version` до 200 или таймаут.
  - `navigateToUrl(url, deps)` — навигация по URL с записью в lastUrlState; требует работающий Chrome.
  - `readLastUrl`, `writeLastUrl` (lastUrlState) — чтение/запись последнего URL из файла.
  - Нет явного API «проверить, жив ли Chrome» и «рестартовать Chrome» (убить процесс + запустить + открыть last URL или idle).

- **OBS**
  - `createObsModule(config, logger)` — возвращает `AppModule` с `start()`; внутри после запуска вешается `proc.once('exit', ...)` и при выходе делается рестарт с задержкой `getRestartDelayMs`.
  - `getObsProcess(): ChildProcess | null` — текущий процесс OBS.
  - `launchObs(obsPath, args, timeoutMs, logger)` — запуск и ожидание ready (внутри модуля).
  - `getRestartDelayMs(lastRestartAt, minIntervalMs)`, `shouldThrottleRestart` — в `restart.ts` для ограничения частоты рестартов.
  - Нет публичного API «рестартовать OBS» по вызову извне (рестарт только по обработчику `exit`).

- **Orchestrator**
  - `runOrchestrator(modules, logger)` — последовательно вызывает `start()` у каждого модуля и завершается; не держит цикл проверок.

- **Config**
  - `validateEnv(): AppConfig` — читает и валидирует env; возвращает типизированный конфиг.
  - В конфиге нет полей для интервала проверки watchdog и (при необходимости) для интервала/лимитов backoff рестартов.

## Зависимости и ограничения

- **Зависимости:** модули Chrome (Этап 2), переключение URL и lastUrlState (Этап 3), модуль OBS (Этап 4). Idle-сервер и страница уже есть (Этап 1).
- **Ограничение частоты рестартов:** в OBS уже есть backoff внутри модуля (`RESTART_MIN_INTERVAL_MS`, `getRestartDelayMs`). Для watchdog нужна единая или согласованная политика: общий лимит частоты рестартов и/или отдельные интервалы для Chrome и OBS — уточняется при реализации; интервал **проверки** статуса обязательно настраивается через env.
- **Риски:** при рестарте Chrome нужно корректно завершить старый процесс (если ещё жив) и не конфликтовать с портом DevTools; при рестарте OBS — не создавать дублирующих подписок на `exit` или обеспечить единую точку рестарта.
- **Внешние зависимости:** конфигурация только из переменных окружения (без файлов конфига); новые параметры должны проходить валидацию в модуле config.
