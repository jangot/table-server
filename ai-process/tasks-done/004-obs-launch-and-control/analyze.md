# Анализ: Запуск и контроль OBS

## Общее описание функциональности

Нужно реализовать **отдельный модуль** только для OBS: запуск OBS после готовности Chrome (по вызову оркестратора), запуск с сохранённой конфигурацией/сценой (путь из env), определение готовности (процесс и при необходимости порт), при падении — перезапуск с ограничением частоты рестартов и логированием причины. Логика Chrome и других приложений в этот модуль не входит. Порядок «сначала Chrome, потом OBS» обеспечивается оркестратором: оркестратор вызывает `start()` у модуля Chrome, дожидается разрешения Promise, затем вызывает `start()` у модуля OBS.

Задача решает этап 4 плана: после поднятия Chrome сервис должен поднять OBS с нужной сценой и уметь перезапускать OBS при падении без бесконечных рестартов.

## Связанные модули и сущности

| Модуль/файл | Назначение | Что затрагивает задача |
|-------------|------------|-------------------------|
| `src/modules/orchestrator` | Последовательный запуск приложений по контракту `AppModule` | Модуль OBS должен реализовать `AppModule` и подставляться вместо заглушки |
| `src/modules/chrome` | Запуск Chrome, ожидание DevTools, экспорт «готов» через разрешение `start()` | Не меняется; оркестратор уже ждёт готовности Chrome перед следующим модулем |
| `src/modules/config` | Типы и валидация env, типизированный конфиг | Добавить поля для OBS: опционально путь к конфигурации/профилю/сцене, при необходимости — параметры ограничения рестартов |
| `src/modules/startup-checks` | Проверка наличия исполняемых Chrome и OBS | Не меняется; уже проверяется `config.obsPath` |

## Текущие интерфейсы и API (если есть)

- **`AppModule`** (`src/modules/orchestrator/index.ts`): `{ name: string; start(): Promise<void> }`. Метод `start()` должен завершиться (Promise resolve) только после готовности приложения, чтобы оркестратор перешёл к следующему модулю. Модуль OBS должен реализовать этот интерфейс.
- **Оркестратор** вызывает модули по порядку: `await mod.start()` для каждого. Текущий порядок в `src/index.ts`: Chrome, затем заглушка OBS.
- **Конфиг** (`AppConfig`): уже есть `obsPath: string`, `obsReadyTimeout?: number`. В `validate.ts` читаются `OBS_PATH`, `OBS_READY_TIMEOUT`. Пути к конфигурации/профилю/сцене OBS в конфиге и env пока не описаны — при необходимости добавить опциональные переменные (например, каталог профиля OBS или путь к сцене, если OBS поддерживает это через CLI).
- **Модуль Chrome** — образец реализации: `createChromeModule(config, logger)` возвращает `AppModule`; внутри `start()` вызывается `launchChrome(...)`, который spawn’ит процесс и через `waitForDevTools` ждёт готовности, затем resolve. Для OBS аналог: spawn OBS, дождаться «готовности» (процесс жив и при необходимости дополнительный критерий — порт/файл), затем resolve.
- **Запуск процесса**: в Chrome используется `child_process.spawn(path, args, { stdio: 'ignore', shell: false })` — без shell, чтобы избежать command injection. Для OBS — тот же подход: только путь и аргументы из конфига/валидированных значений, без подстановки пользовательского ввода.

## Зависимости и ограничения

- **Оркестрация:** порядок запуска задаётся в `src/index.ts` (сначала Chrome, потом OBS). Модуль OBS не должен вызывать или зависеть от модуля Chrome; оркестратор гарантирует, что OBS стартует только после разрешения Promise от Chrome.
- **Платформа:** целевая ОС — Linux; пути к OBS и опции запуска (профиль/конфиг) ориентированы на Linux. OBS может не иметь единого CLI для «загрузить сцену X» — в таком случае достаточно запуска с путём к каталогу конфигурации/профиля, если он поддерживается, иначе использование конфига по умолчанию (требование: «OBS с заранее сохранённой конфигурацией» выполняется настройкой окружения/профиля до запуска сервиса).
- **Готовность OBS:** в отличие от Chrome, у OBS нет DevTools endpoint. Критерий готовности — процесс запущен и (опционально) порт или другой индикатор, если применимо. Документация OBS (например, obs-websocket) не обязательна для MVP — достаточно проверки процесса и таймаута.
- **Падение и рестарт:** обработка события `exit` у процесса OBS, логирование кода/причины, повторный запуск с ограничением частоты (например, не чаще чем раз в N секунд и/или экспоненциальный backoff), чтобы избежать бесконечных циклов. Детали политики рестартов могут быть в конфиге (env) или зафиксированы в модуле.
- **Этап 5 (watchdog):** модуль OBS должен предоставлять способ проверки «запущен ли OBS» и при необходимости «перезапустить» для использования в watchdog — либо экспорт функций/геттеров из модуля OBS, либо единый контракт модулей приложений (если будет введён на этапе 5). В рамках этапа 4 достаточно, чтобы перезапуск при падении и ограничение частоты были реализованы внутри модуля OBS.
- **Безопасность:** запуск OBS только через `spawn` с массивом аргументов, без конкатенации пользовательского ввода в команду shell (как в модуле Chrome).
- **Тесты:** в `test/` есть тесты для config, chrome, idle-server; при реализации модуля OBS имеет смысл добавить тесты (например, конфиг с путём OBS, мок процесса для проверки логики готовности и рестарта).
