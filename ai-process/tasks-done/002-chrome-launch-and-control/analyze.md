# Анализ: Запуск и контроль Chrome

## Общее описание функциональности

Нужно реализовать **отдельный модуль** для работы с Chrome: запуск процесса с remote debugging, ожидание готовности DevTools, поддержка одного окна/сессии. Модуль должен реализовывать контракт оркестратора (интерфейс `AppModule`): метод `start()` возвращает `Promise<void>`, который резолвится только после готовности Chrome, чтобы оркестратор мог запускать следующее приложение (OBS) только после этого. Логика OBS и других приложений в этот модуль не входит.

Задача решает этап 2 плана выполнения: после этапа 1 (оркестрация и конфигурация) подменить заглушку Chrome реальным запуском браузера и дать оркестратору сигнал «ready» через разрешение промиса `start()`.

## Текущие интерфейсы и API (если есть)

- **`AppModule`** (orchestrator):
  - `name: string`
  - `start(): Promise<void>` — должен резолвиться, когда приложение готово (для Chrome — процесс запущен и DevTools endpoint доступен).

- **`AppConfig`** (config/types.ts):
  - `chromePath: string` — путь к исполняемому Chrome;
  - `devToolsPort?: number` — порт remote debugging (опционально);
  - `chromeReadyTimeout?: number` — таймаут ожидания готовности (мс);
  - `idlePort: number` — порт idle-сервера (URL для стартовой страницы: `http://localhost:<idlePort>/`).

- **`Logger`** (logger): `info`, `warn`, `error`, `debug`. Оркестратор не передаёт логгер в `start()` — значит, модуль Chrome нужно создавать с зависимостями (config, logger) и экспортировать фабрику `createChromeModule(config, logger): AppModule`.

- Режим окна (kiosk/app/fullscreen) в конфиге **не описан**; в требованиях и плане указано «режим окна — kiosk/app/fullscreen из конфигурации». Значит, конфиг и валидацию env при необходимости расширить (например, опциональная переменная `CHROME_WINDOW_MODE`).

## Зависимости и ограничения

- **Зависимость от этапа 1:** конфиг, оркестратор, проверка наличия Chrome, idle-сервер уже есть. Пути и порты берутся из конфигурации (env). Idle-сервер к моменту вызова оркестратора уже запущен, URL idle-страницы — `http://localhost:{config.idlePort}/`.
- **Безопасность:** запуск Chrome только через `child_process.spawn` с массивом аргументов, **без shell** (никакой пользовательский ввод в командную строку не подставлять), защита от command injection.
- **Порт DevTools:** в конфиге `devToolsPort` опциональный. Для работы модуля Chrome порт нужен; либо задать значение по умолчанию (например 9222), либо считать обязательным при наличии модуля Chrome (валидация/документация).
- **Одно окно/сессия:** запуск с одним URL (idle), без лишних вкладок; при необходимости — выбор target по CDP и сохранение последнего URL в состоянии (задача 003 может опираться на это).
- **Внешние зависимости:** только стандартные модули Node.js (`child_process`, `http`/`https` или `net` для проверки порта) — без новых npm-пакетов при желании; при необходимости можно использовать библиотеку для HTTP-запросов.
- **Риски:** занятость порта DevTools другим процессом; таймаут готовности — обработать ошибки и логировать, при неудаче — отклонить Promise (оркестратор не перейдёт к OBS).
