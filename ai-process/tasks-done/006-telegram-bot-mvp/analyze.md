# Анализ: Внешний интерфейс — Telegram-бот (MVP)

## Общее описание функциональности

Нужно реализовать приём команд через Telegram Bot API: минимум — команда с URL для открытия в уже запущенном Chrome. Оператор присылает ссылку боту → бот проверяет, что отправитель в списке разрешённых пользователей → вызывает логику переключения URL (Этап 3) → возвращает подтверждение или ошибку (система не готова, Chrome/OBS недоступны). Неавторизованные запросы отклоняются с логированием.

Задача закрывает этап 6 плана выполнения и даёт оператору возможность управлять отображаемой страницей через Telegram без изменения кода при последующей подмене источника списка пользователей на БД.

## Связанные модули и сущности

| Модуль/файл | Назначение | Что затрагивает задача |
|-------------|------------|-------------------------|
| `src/modules/config` | Валидация env, типизированный конфиг | Добавить поля: токен бота, список разрешённых пользователей (имена/логины или id). |
| `src/modules/users` | **Новый модуль.** Работа с разрешёнными пользователями. | Создать с абстракцией над источником списка; в MVP источник — env. Интерфейс должен позволять позже подменить на БД без изменения кода бота. |
| `src/modules/chrome` | Запуск Chrome, навигация по URL через CDP | Вызов `navigateToUrl` из бота; проверка готовности через `isChromeAlive`. |
| `src/modules/obs` | Запуск OBS | Проверка готовности через `isObsAlive` для сообщений об ошибках («система не готова»). |
| `src/modules/logger` | Логирование | Логировать входящие команды и отказы неавторизованным. |
| `src/index.ts` | Точка входа, оркестрация | Запуск бота после оркестратора (Chrome и OBS уже подняты); передача в бот зависимостей (config, logger, users, функции навигации и проверки живости). |
| Telegram Bot API | Внешний сервис | Новый модуль бота: long polling или webhook, парсинг сообщений, отправка ответов. |

## Текущие интерфейсы и API (если есть)

- **`AppConfig`** (`src/modules/config/types.ts`) — сейчас полей для бота нет; в комментарии указано «Extended in later stages: botToken, allowedUsers».
- **`validateEnv()`** (`src/modules/config/validate.ts`) — возвращает `AppConfig`; нужно добавить чтение и валидацию `TELEGRAM_BOT_TOKEN` и списка разрешённых пользователей (например `ALLOWED_TELEGRAM_USERS` — строка с разделителем).
- **`navigateToUrl(url, deps: { config, logger })`** (`src/modules/chrome/index.ts`) — асинхронная навигация по URL; при не запущенном Chrome выбрасывает `Error('Chrome is not running')`. Вызывается из бота при успешной авторизации и наличии URL в команде.
- **`isChromeAlive(config)`**, **`isObsAlive(config)`** — экспортируются из `src/index.ts` и передаются в watchdog; боту нужны аналогичные проверки для формирования ответов «система не готова».
- **`Logger`** (`src/modules/logger`) — интерфейс `info`, `warn`, `error`, `debug`; используется по всему коду.
- **Модуля пользователей и бота в коде пока нет** — только упоминания в плане и требованиях.

## Зависимости и ограничения

- **Зависимость от этапов 1, 3:** конфигурация (в т.ч. токен и пользователи) и переключение URL уже реализованы. Бот только добавляет канал ввода и вызова существующей логики.
- **Порядок запуска:** бот должен стартовать после оркестратора, чтобы к моменту первой команды Chrome и OBS были запущены. Ответы «система не готова» или «Chrome/OBS недоступны» — при сбоях после старта (watchdog ещё не перезапустил).
- **Формат идентификации пользователей Telegram:** в MVP список из env может содержать Telegram user id (число) и/или username (строка без `@`). Нужно определиться с форматом одной переменной (например запятые, поддержка и id и username).
- **Валидация URL:** в MVP не требуется (по плану); бот передаёт строку в `navigateToUrl` как есть.
- **Библиотека для Telegram:** выбор между `telegraf` и `node-telegram-bot-api`; у обеих есть типы и поддержка long polling. Webhook в MVP не обязателен.
- **Риски:** при пустом или неверном токене бот не должен падать весь процесс — обработать ошибку запуска и логировать; опционально сделать запуск бота условным (если токен не задан — не стартовать бота).
- **Тесты:** в проекте используются `node:test` и моки env; для модуля users и бота потребуются юнит-тесты (users — по списку из конфига; бот — при необходимости мок Telegram API или интеграционные тесты).
