# План реализации: Автозапуск и процесс-менеджер (PM2)

Подготовить документацию и пример конфигурации PM2 для Linux: запуск сервиса при входе в графическую сессию, рестарт при падении, сохранение логов. При необходимости — краткая инструкция по автологину и запуску PM2 после входа. Код приложения не меняется; результат — файлы в `docs/` и пример ecosystem-файла.

## 1. Пример конфигурации PM2 (ecosystem)

**Файл (создать):** `ecosystem.config.cjs` (в корне репозитория)

Создать пример конфига PM2 в формате CommonJS (чтобы не зависеть от `"type": "module"` в package.json). Включить:

- `name` — имя приложения (например, `table-server`).
- `script` — путь к скомпилированному входу: `dist/index.js` (после `npm run build`).
- `cwd` — рабочая директория приложения (корень проекта).
- `autorestart: true`, при необходимости `max_restarts` и `min_uptime`.
- `env` — пример переменных или отсылка к `.env`; не хранить секреты в репозитории.
- `env_file` — путь к `.env` (если PM2 поддерживает в используемой версии), иначе описать в документации передачу env через ecosystem или загрузку вручную.
- `out_file` и `error_file` — пути для логов (например, `logs/out.log`, `logs/error.log`) или оставить стандартные места PM2 с пояснением в документации.
- Комментарии в файле на английском: что подставлять, где хранить секреты, как запускать.

```javascript
// ecosystem.config.cjs
// Copy to your deployment directory and adjust paths/env. Do not commit secrets.

module.exports = {
  apps: [{
    name: 'table-server',
    script: 'dist/index.js',
    cwd: '/path/to/table-server',
    autorestart: true,
    env: {
      NODE_ENV: 'production',
      // CHROME_PATH, OBS_PATH, IDLE_PORT, etc. — set here or via env_file
    },
    // env_file: '.env',  // if supported by your PM2 version
    out_file: 'logs/out.log',
    error_file: 'logs/error.log',
  }],
};
```

При отсутствии поддержки `env_file` в выбранной версии PM2 — в документации описать использование `env` в ecosystem или загрузку `.env` перед `pm2 start`.

## 2. Документ по развёртыванию с PM2 и автозапуском

**Файл (создать):** `docs/deployment-pm2.md`

Создать документ на английском (в соответствии с правилами проекта для документации). Разделы:

1. **Предварительные условия**  
   Node.js, Chrome, OBS установлены; приложение собрано (`npm run build`). Ссылка на `docs/env.md` для переменных окружения.

2. **Установка PM2**  
   Глобально: `npm install -g pm2` (указать рекомендуемую или минимальную версию, если важно). Альтернатива: запуск через `npx pm2` с указанием рабочей директории.

3. **Запуск приложения через PM2**  
   Скопировать/адаптировать `ecosystem.config.cjs`, задать `cwd`, пути логов, переменные окружения. Команды: `pm2 start ecosystem.config.cjs`, `pm2 status`, `pm2 logs`, `pm2 restart table-server`. Указать, что команда для запуска — скомпилированный `dist/index.js` (соответствует `npm run start` после сборки).

4. **Автозапуск при входе в сессию (Linux)**  
   Целевой способ: `pm2 startup` (пользовательский режим), затем `pm2 save`. Явно указать, что речь о запуске после входа пользователя в графическую сессию (не headless). Кратко описать шаги: выполнить вывод команды `pm2 startup`, сохранить дамп через `pm2 save`. При необходимости упомянуть альтернативы (systemd user unit, xdg-autostart) без детализации.

5. **Логи**  
   Где PM2 хранит логи (out/error); как настроить пути в ecosystem; ротация логов (ссылкой на документацию PM2 или кратко).

6. **Секреты**  
   Не коммитить токены в ecosystem; использовать `.env` в рабочей директории (в .gitignore) или переменные окружения процесса; при использовании env в ecosystem — не хранить секреты в репозитории.

7. **Опционально: автологин в Linux**  
   Краткая инструкция или ссылка: как включить автологин пользователя в графическую сессию (зависит от дистрибутива: GDM, LightDM и т.д.), с напоминанием, что после входа должен выполняться автозапуск PM2 (п. 4). Либо один общий параграф с предупреждением о различии дистрибутивов и ссылкой на внешнюю документацию.

Язык раздела — английский.

## 4. Проверка (чеклист)

**Файл:** не создаётся; сценарии проверки описать в `docs/deployment-pm2.md` в виде краткого раздела «Verification» или включить в конец документа.

Сценарии для ручной проверки после выполнения плана:

- **Happy path:** по инструкции из `deployment-pm2.md` установить PM2, скопировать и подставить пути в `ecosystem.config.cjs`, выполнить `pm2 start`; убедиться, что приложение стартует, в логах есть запись; выполнить `pm2 restart` — приложение перезапускается; после настройки `pm2 startup` и `pm2 save` перезагрузить машину, войти в графическую сессию — процесс table-server поднимается без ручного запуска.
- **Ошибки:** при неверном `cwd` или отсутствии `dist/index.js` PM2 логирует ошибку; при отсутствии обязательных переменных окружения приложение падает с понятным сообщением (как уже реализовано в коде).

Отдельный тестовый файл (unit/integration) для документации и конфига не требуется; достаточно чеклиста в документе.
