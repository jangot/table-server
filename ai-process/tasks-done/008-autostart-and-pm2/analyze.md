# Анализ: Автозапуск и процесс-менеджер (PM2)

## Общее описание функциональности

Задача — подготовить **документацию и примеры конфигов** под PM2 на Linux, чтобы:

- Сервис запускался при старте пользовательской сессии (после входа в графическую среду).
- При падении процесса сервиса PM2 выполнял рестарт.
- Логи сохранялись (стандартные выходы и ошибки под управлением PM2).

При необходимости — дать минимальный скрипт или инструкцию для **автологина в Linux** и запуска сервиса через PM2 после входа пользователя.

Код приложения менять не требуется: целевой результат — инфраструктурная документация и примеры конфигурации (ecosystem file, автозапуск, при необходимости автологин). Контекст: `docs/plan-execution.md`, Этап 8; решение по процесс-менеджеру — PM2 (раздел 1.4).

## Текущие интерфейсы и API (если есть)

- **Запуск приложения:** `npm run start` (сборка `tsc` + `node dist/index.js`) или после сборки — `node dist/index.js`. Для разработки — `npm run dev` (ts-node).
- **Конфигурация:** только переменные окружения; при старте загружается `.env` из текущей рабочей каталога (если есть), поверх — переменные процесса. Валидация в `getConfig()`; при ошибке — вывод в stderr и `process.exit(1)`.
- Специфичных API для PM2 или автозапуска в коде нет — задача решается конфигами и документацией.

## Зависимости и ограничения

- **ОС:** целевая платформа — Linux (пути, автозапуск в сессии пользователя, при необходимости автологин).
- **Графическая сессия:** сервис должен стартовать после входа пользователя в графическую сессию (не headless), т.к. от неё зависят Chrome и OBS.
- **PM2:** внешняя зависимость; в документации нужно описать установку (глобально или через npx) и версию по возможности.
- **Порядок запуска:** до старта сервиса должны быть доступны Chrome, OBS и настроенное окружение (пути в env); приложение уже проверяет наличие исполняемых файлов при старте (`checkChromeAndObs`).
- **Секреты:** в `docs/env.md` указано не класть секреты в коммитируемые файлы; в примере ecosystem — использовать `env_file` или переменные окружения, не хранить токены в репозитории.
- **Риски:** разные дистрибутивы Linux и способы автозапуска при входе (autostart, systemd user, xdg-autostart и т.д.) — документация должна явно указать целевой способ (например, автозапуск при входе в сессию через PM2 startup + сохранение сгенерированного скрипта) и при необходимости упомянуть альтернативы.
