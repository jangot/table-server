# Задача: Привязка Chrome-окна к источнику OBS при старте

## Суть задачи

При старте системы необходимо автоматически определить X11 Window ID запущенного Chrome
и передать его в OBS через WebSocket, чтобы источник Window Capture (Xcomposite) захватывал
правильное окно. Без этого OBS либо показывает пустой экран, либо захватывает устаревший XID
после перезапуска Chrome.

## Детали и требования

- Определять X11 Window ID через `xdotool search --onlyvisible --class chrome`
- Реализовать retry-логику: опрашивать каждые 500 мс, таймаут 10 секунд
- После успешного определения XID вызывать `setInputSettings` на OBS клиенте с именем источника
- Имя источника в OBS задаётся через конфиг (`obs.chromeSourceName`); если не задано — фича отключена
- Источник типа Window Capture (Xcomposite) должен существовать в OBS заранее (создаётся вручную)
- В документацию (README или отдельный docs-файл) добавить раздел о ручной настройке OBS:
  - как создать источник Window Capture (Xcomposite) в нужной сцене
  - что имя источника должно совпадать с `OBS_CHROME_SOURCE_NAME` в `.env`
  - что привязка к окну происходит автоматически при каждом старте сервиса
  - что без этого шага фича не работает
- Порядок старта в оркестраторе сохраняется: Chrome → OBS → привязка
- Привязку вызывать через `onConnected` callback OBS WebSocket клиента

## Контекст

- Требование описано в `docs/requirements/obs_chrome_window_binding.md`
- `ObsWebSocketClient.setInputSettings()` уже реализован в `src/modules/obs-scenes/client.ts:227`
- Chrome запускается через `src/modules/chrome/launch.ts`, ждёт DevTools перед резолвом
- OBS запускается после Chrome в `src/index.ts` через `runOrchestrator`
- Системный пакет: `xdotool` (`apt install xdotool`)

## Риски

- **X11 only** — подход работает исключительно на X11. На Wayland `xdotool` не работает.
  Если система переходит на Wayland, потребуется альтернативный механизм (например, через PID окна).
- **Пересоздание окна Chrome** — Chrome может пересоздать окно (GPU reset, fullscreen toggle,
  compositor changes). В этом случае старый XID становится невалидным, и OBS снова теряет источник.
  Для надёжности нужен watchdog, отслеживающий валидность XID и перепривязывающий при необходимости.
  В рамках данной задачи watchdog не реализуется — фиксируется как известное ограничение.
- **Гонка при старте** — если Chrome очень быстро пересоздаёт окно сразу после запуска,
  xdotool может вернуть временный XID. Retry-логика частично снижает этот риск.
- **Отсутствие источника в OBS** — если источник `chromeSourceName` не существует в OBS,
  вызов `setInputSettings` вернёт ошибку. Необходимо перехватывать и логировать, не падать.
